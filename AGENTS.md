# HEYS v2 — Agent Notes

> Этот файл задаёт предпочтительный стиль работы агентов в репозитории. Он **не
> отменяет** обязательные правила из `.github/copilot-instructions.md`, особенно
> для legacy runtime, rebuild и browser verification в runtime-sensitive
> случаях.

## Core intent

- Считай `.github/copilot-instructions.md` источником operational/default
  правил, а этот файл — источником более мягких предпочтений по стилю работы.
- Предпочитай практичные и локальные правки вместо больших необязательных
  рефакторингов.
- Сначала решай задачу пользователя, потом улучшай окружающий код только если
  это действительно помогает.
- Не воспринимай этот файл как набор жёстких ограничений: если более надёжная
  реализация требует другого подхода, выбери её и кратко объясни почему.

## UI and frontend workflow

- Для operational-решений по rebuild, browser verification, legacy bundle
  validation и уровню проверки ориентируйся прежде всего на
  `.github/copilot-instructions.md`.
- Для low-risk задач вне legacy runtime и вне runtime-sensitive поведения
  предпочитай быстрый путь без браузера.
- Пользователь часто сам проверяет визуальный результат, поэтому без явной
  пользы не трать время и токены на browser verification.
- Этот файл не запрещает использовать браузер, новые компоненты, хуки, утилиты
  или другие техники, если они действительно нужны для качественного результата.

## Legacy/runtime caution

- Если правка затрагивает `apps/web/**` и влияет на legacy runtime bundle,
  следуй более строгому пути: rebuild, файловая валидация и browser
  verification, когда этого требуют правила репозитория.
- Не ослабляй требования к post-build validation только потому, что задача
  визуально кажется небольшой.
- Если есть сомнение, относится ли изменение к runtime-sensitive, безопаснее
  следовать более надёжному пути.

## Implementation freedom

- Не ограничивай себя текущим локальным паттерном, если он мешает ясному и
  надёжному решению.
- Можно создавать новые компоненты, модули, утилиты и конфиги, если это
  соразмерно задаче и реально упрощает поддержку.
- Избегай «идеологического» переписывания кода без прямой пользы для текущей
  задачи.

## Communication

- В финале кратко сообщай:
  - что изменено;
  - что проверено;
  - какая валидация была выполнена: файловая или файловая + браузерная;
  - если browser verification была осознанно пропущена — почему это было
    безопасно.
- Если выбран нестандартный путь реализации, коротко зафиксируй причину.
