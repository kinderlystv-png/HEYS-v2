## Главное: сторона прихода чистая и консистентная

1. **Нет ни одного stored `kcal100`** в shared-базе. Продукты хранят только
   компонентные макросы
   (`simple100, complex100, protein100, badFat100, goodFat100, trans100, fiber100`).
   Калории **всегда** выводятся (`computeDerivedProduct`, net 3/4/9). →
   Конфликта «введённый gross kcal vs derived net» по продуктам **не
   существует**. Приход равномерно net.

2. **`kcal100_legacy` — vestigial.** Колонка есть в БД, но (а) не читается нигде
   в live-коде (grep пуст), (б) в выгрузке null у всех. Старого gross-значения,
   которое могло бы протечь, нет.

3. **V-1a (продукт с ккал, но без макросов → 0 ккал) в живой базе НЕ
   проявляется.** После нормализации регистра — **0 продуктов с 0 ккал** в обоих
   экспортах. Edge-case остаётся теоретическим (код-путь существует), но
   реальных данных под него нет. → Понижаю V-1a из «data-проблема» в
   «робастность кода» (Low).

4. **Аномалий нет:** 0 продуктов с макросуммой >100 г/100 г, 0 с kcal>902, 0
   отрицательных. Покрытие полей полное: `protein100`/`gi`/`harm` = 284/284.

## Что мой первый скан показал ошибочно (и почему)

Сырой БД-экспорт хранит жиры в нижнем регистре (`badfat100`, `goodfat100`), а
`computeDerivedProduct` (`models:581`) и мой скан читают camelCase
(`badFat100`). Без нормализации «Подсолнечное масло» вышло 0 ккал,
масло/майонез/сало — резко занижены. **Это артефакт скана, не live-баг:** живой
путь чтения (`getProductFromItem` → `normalizeProductFields`,
`models:729/748/...,621-622`) маппит `badfat100→badFat100` до расчёта. С
нормализацией масло = 899 ккал, как и должно.

> ⚠️ **Латентная хрупкость (P-1, Low):** `computeDerivedProduct` читает только
> camelCase и сам регистр не чинит — корректность ккал держится на том, что
> `normalizeProductFields` вызван **выше по стеку**. Любой новый код-путь,
> передавший сырой DB-продукт в `per100`/`mealTotals`/`computeDerivedProduct`
> минуя нормализацию, молча получит `fat=0` → заниженные ккал. Стоит занести
> нормализацию регистра внутрь `computeDerivedProduct` для надёжности.

## Реальный методологический пробел: этанол

**Алкоголь системно недосчитывается** — этанол даёт 7 ккал/г, но не входит в
Atwater-модель (только белок/угл/жир). В базе:

- «Bakalář пиво» → **15 ккал/100 г** (реально пиво ~40). Недосчёт ~60%.

В каталоге всего ~1 настоящий алкогольный продукт (остальные «совпадения» —
ложные: «Виноград» по подстроке «вино», «сВИНОграбья котлета» и т.п.), поэтому
влияние на общую методологию мизерное. Но как пробел модели — реален: нет
ethanol-терма. Если у клиентов есть алкоголь в дневниках — он занижен.
**Severity Low** (мало продуктов), но отметить.

## Низкокалорийные (контекст, не баги)

13 продуктов <25 ккал/100 г — почти все легитимны: огурец, помидор, кофе
американо (2), квашеная капуста, миндальное молоко без сахара. Единственный

## Вклад белка по реальной базе (ось net 3 vs gross 4)

- Средний белок: **9.9 г/100 г**.
- Доля «белковых» ккал в net-сумме каталога: **~15%**.
- Net-сумма каталога **на ~5% ниже** gross-суммы (4/4/9).

→ Осознанный net-выбор реально снижает логируемый приход на ~5% против этикеток
(4/4/9). Это и есть величина эффекта на живых данных. В связке с багом B-2
(норма делит белок на 4) именно эти ~5% и создают рассинхрон «грамм-цель vs
счётчик».

## Итог по приходу

Сторона прихода **научно когерентна и чиста на данных**: net-Atwater применён
единообразно, stored-kcal конфликта нет, нулевых/битых продуктов нет, поля
покрыты. Реальные мелочи: этанол не моделируется (Low), нормализация регистра —
внешняя зависимость расчёта ккал (P-1, Low). Ни одного High на стороне прихода.

Фундамент (net-приход) подтверждён на реальных данных. Все High-находки аудита —
на стороне **нормы/расхода** (B-2 белок /4, B-3 две формулы BMR), а не прихода.

## SQL для самостоятельного прогона на проде (per-client overlay)

```sql
-- Продукты с макросами, но потенциально 0 ккал (живой V-1a по всем клиентам)
-- (вычисление net-Atwater прямо в SQL по элементам overlay-массива)
SELECT substring(client_id::text,1,8) AS cid8,
       count(*) FILTER (WHERE 3*COALESCE((e->>'protein100')::numeric,0)
                            +4*(COALESCE((e->>'simple100')::numeric,0)+COALESCE((e->>'complex100')::numeric,0))
                            +9*(COALESCE((e->>'badfat100')::numeric,0)+COALESCE((e->>'goodfat100')::numeric,0)+COALESCE((e->>'trans100')::numeric,0)) = 0) AS zero_kcal,
       jsonb_array_length(v) AS total
FROM client_kv_store, jsonb_array_elements(v) e
WHERE k='heys_products_overlay_v2' AND jsonb_typeof(v)='array'
GROUP BY client_id, v
ORDER BY zero_kcal DESC NULLS LAST
LIMIT 30;
```

Запуск: `./scripts/db/psql.sh -f <файл>.sql` (из среды, где доступен
Lockbox/yc).
