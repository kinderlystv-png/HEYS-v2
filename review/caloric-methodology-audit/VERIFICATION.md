# Верификация открытых мест (закрытие неуверенностей)

Дата: 2026-06-09. Прогон по всем пунктам, где в 1–3 проходах стояло «надо
подтвердить». Калорийность — фундамент приложения, поэтому каждый закрыт чтением
кода, а не предположением. Ниже — что проверено и что изменилось в выводах.

## V-1 · Приход всегда net(белок×3), сырой kcal100 игнорируется — ✅ ПОДТВЕРЖДЕНО

Цепочка: `mealTotals` (`heys_models_v1.js:838`) → `per100` (`:782`) →
`computeDerivedProduct` (`:577-583`). kcal **всегда** пересчитывается из
макросов:

```
kcal = 3 × protein100 + 4 × carbs + 9 × fat   // models:581
```

Введённый/импортированный `kcal100` на продукте в подсчёт дня **не попадает** —
берётся только derived. Значит приход консистентно net по всем продуктам, без
смеси gross/net. Линза B (`eaten_net − tdee`) опирается на это корректно.

> ⚠️ Побочный edge-case (новый, V-1a, **Low**): продукт с заполненным `kcal100`,
> но **без макросов** (protein/carbs/fat = 0) даст `kcal = 0` в `mealTotals`.
> «Только калории»-продукты (частый кейс импорта/быстрого ввода) обнулятся в
> дневном итоге. Стоит проверить на реальной базе продуктов, нет ли таких строк.

## V-2 · Weekly «плановый дефицит %» (B-1) — ✅ протечки на практике НЕТ → снижено до Low

`resolveTargetDeficitPct` (`heys_day_insights_data_v1.js:12-22`) возвращает
дефицит по приоритету: `day.deficitPct` → `tdeeInfo.deficitPct` →
`profile.deficitPctTarget` → и лишь последним fallback'ом
`(goal−burned)/burned`. Так как `deficitPctTarget` по умолчанию `0` (а это
finite), функция почти всегда отдаёт **сконфигурированный** дефицит, а смешанная
формула фактически недостижима. Инлайновый смешанный `targetPctFromGoal` в
`weekly:215` — используется только если `resolveTargetDeficitPct` отсутствует (а
он определён).

**Вывод:** B-1 — мёртвый код / теоретическая протечка, не реальный баг. Понижаю
Medium → **Low**. Фактический недельный delta% (`weekly:234`,
`(totalKcal − totalBurned)/totalBurned`) — чистая Линза B, корректен.

## V-3 · Cycle-мультипликатор — ✅ применяется ОДИН раз, двойного счёта нет

- Применяется к таргету ровно один раз: `heys_tdee_v1.js:215`
  `optimum = baseOptimum × cycleKcalMultiplier`.
- В балансе (`heys_day_caloric_balance_v1.js:918-931`) `getKcalMultiplier`
  используется **только для advisory-текста** «🌸 норма увеличена на X%», к
  дельте/долгу не применяется.

Двойного наложения cycle × cycle нет. (Согласуется с выводом 2-го прогона про
adaptive thermogenesis — тоже advisory.)

## V-4 · Календарь vs live optimum — расхождение шире, чем только NDTE

Уточнение к F-07. `getActiveDaysForMonth` (`heys_day_utils.js:1842`):
`target = tdee × (1 + deficit)`, где
`tdee = bmr + steps + household + trainings`. Live `optimum`
(`tdee:201,214-215`) = `(bmr + act + ndteBoost) × (1+deficit) × cycleMult`.
Значит для **не-просмотренных** дней календарный target расходится с тем, что
было бы живьём, на:

1. **NDTE-буст** (если вчера тренировка ≥200 ккал), и
2. **cycle-мультипликатор** (для женщин с трекингом цикла, ~5–10%).

Митигировано: для просмотренных дней берётся `savedDisplayOptimum`
(`day_utils:1846`), т.е. реальное показанное значение. Расхождение только на
днях без сохранённого снимка. Severity F-07 остаётся **Low**, но в заметке
теперь указан и cycle-фактор, не только NDTE.

## V-5 · Главное число дефицита на герое = Линза B (с TEF) — ✅ ПОДТВЕРЖДЕНО

`heys_day_main_block_v1.js:204`: `deficitKcal = eatenKcal − tdee` (tdee с TEF).
И «расход» в таблице (`:61`) = `tdee`. То есть основной дневной
дефицит/профицит, который видит пользователь, — это физиологически корректная
Линза B. Долг/норма (Линза A) — отдельный компенсаторный механизм. Дуальность
реальна и осознана.

## V-6 · `getDayTdee` — ✅ висячая ссылка, 4-го пути TDEE НЕТ

`dayUtils.getDayTdee` **нигде не определён** (проверено grep'ом по всему
`apps/web`, исключая бандлы). Все вызовы (`weekly:208`, `insights_data:493,665`,
`cascade_card:992`) имеют fallback на `HEYS.TDEE.calculate` или guard
`typeof === 'function'`. Значит недельный/каскадный `burned` идёт через
канонический модуль с TEF — отдельной 4-й методики расчёта нет. (Висячую ссылку
стоит вычистить, но на числа не влияет.)
