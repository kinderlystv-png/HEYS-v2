## 1. Почему скролл плавный на iPhone и рваный на P30 — подтверждённые причины

Ранжированы по ожидаемому вкладу.

### S1. Постоянный frosted-glass на шапке Дня: `backdrop-filter: blur(12px) saturate(1.4)`

`styles/modules/730-widgets-dashboard.css:183-189` (light) и `:230-236` (dark):
`body:has(.page-day) .hdr { background: rgba(228,241,255, .94); backdrop-filter: blur(12px) saturate(1.4); will-change: transform; }`
— то же для вкладки виджетов.

Шапка зафиксирована над скроллируемым контентом → **каждый кадр скролла** Chrome
пересэмплирует и блюрит полосу контента под ней. WebKit/Apple GPU делает это
почти бесплатно (поэтому iPhone «суперплавный»), Blink/Mali-G76 — дорого, это
хрестоматийная причина рваного скролла на Android. Бонус-наблюдение: селектор
`body:has(.page-day)` заставляет перематчивать `:has()` на мутациях body.

**Фикс — единственный пункт всех 4 прогонов с микровизуальным компромиссом.**
Кэшировать backdrop-blur средствами CSS нельзя — он обязан сэмплировать живой
контент. Варианты:

- **S1-a (рекомендую):** фон шапки уже 94% непрозрачный — blur влияет лишь на
  ~6% пропускания. Заменить на `rgba(..., 0.97)` без blur → на глаз почти
  неотличимо, на Mali экономия единицы-десятки мс/кадр. Под флагом
  `heys_perf_header_glass`, A/B скриншоты light/dark на решение владельца.
- **S1-b:** оставить как есть на iOS/десктопе, solid — только на Android
  (UA-ветка в CSS-классе). Тот же визуальный вопрос, но iPhone не трогаем.
- **S1-c:** ничего не менять (если замер S2/прочих фиксов даст достаточно).

### S2. Sticky meal-bar: forced layout всех карточек на каждом кадре скролла

`heys_day_meals_bundle_v1.js:5135-5161`: window scroll (passive, rAF-throttle —
это хорошо), но в каждом rAF-тике: `querySelectorAll('.meal-card')` + цикл
`getBoundingClientRect()` **по всем карточкам** + `setCurrentIdx` (React-рендер
при смене активной). На дне с 8–12 приёмами = принудительный layout 3–8 мс
каждый кадр скролла, прямо в скролл-пути.

**Фикс (визуально идентичен):** IntersectionObserver с `rootMargin`,
соответствующим STICKY_BAR_LINE (y=100), вместо измерений в скролле — ноль
main-thread работы в кадре; либо кэш offsetTop карточек с инвалидацией на
resize/мутации списка. Активная карточка определяется так же.

### S3. ~500 правил `:hover` без `(hover:hover)`-гарда (защищено только 5)

Подсчёт по styles/: 104 в `000-base-and-gamification.css`, 98 в `fingers.css`,
85 в `heys-components.css`, 74+67+59 в остальных топ-модулях. Многие — с
`transform`/`box-shadow` на строках и карточках (пример:
`400-water-and-hydration.css:1975` —
`tr:hover { transform: translateY(-2px); box-shadow: … }`).

На Android тап даёт sticky-hover: style recalc на 1.7 МБ CSS на каждый тап +
«залипшая» подсветка до следующего тапа. Бьёт по отклику тапа и старту скролла
после тапа.

**Фикс:** обернуть hover-правила в `@media (hover: hover)`. Технически меняет
поведение подсветки по тапу на Android (сейчас она появляется как артефакт
десктопных стилей) → **формально визуальное изменение, на решение владельца**.
Де-факто это чинит «залипшие» подсветки.

### S4. Цена растеризации строк при флинге

Glass-карточки дневника: тень `0 10px 30px -20px` (`heys-components.css:2104`) +
border-radius + полупрозрачности. При быстром флинге Mali растеризует новые
тайлы медленно → дропы кадров. Усилитель, не корень. **Фикс без визуальных
изменений:** `contain: layout style paint` на `.meal-card`/строки (паттерн
contain уже используется — 47 правил), тени НЕ трогать.

### S5. Связка с находками прошлых прогонов (уже в плане)

Фоновые infinite-анимации бара (43 шт.) и shimmer по `left` инвалидируют слои во
время скролла (→ A4/A5); reconcile каждые 3с попадает серединой во флинг =
разовый фриз 10–30 мс (→ A1). Эти фиксы напрямую улучшают скролл.

## 3. Дополнения к плану (PASS2 §4)

### Блок A (визуал идентичен, нулевой риск)

- **A10 (S2).** Sticky meal-bar → IntersectionObserver/кэш offsetTop вместо
  measure-в-каждом-кадре. `heys_day_meals_bundle_v1.js:5135-5161`. Проверка:
  активная карточка переключается на тех же позициях (видео-сравнение),
  скролл-замер до/после. Откат: флаг `heys_disable_io_stickybar`.
- **A11 (S4).** `contain: layout style paint` на карточки/строки дневника.
  Скриншот-дифф обязателен (contain меняет рендер при выступающих тенях —
  проверить, что тень карточки не клипается; если клипается →
  `contain: layout style` без paint).

### Блок D — микровизуальные (решения владельца приняты 2026-06-13)

- **D1 (S1). РЕШЕНО: вариант S1-b — solid только на Android.** iPhone/десктоп
  остаются с glass (blur 12px), Android получает `rgba(..., 0.97)` без blur.
  Реализация: класс `html.heys-android` (UA-детект при буте) + override-правило;
  флаг `heys_perf_header_glass` для отката на blur. Обязателен скриншот-дифф
  light/dark на Android до мержа. Ожидание: самый большой одиночный выигрыш
  плавности скролла на Mali.
- **D2 (S3). РЕШЕНО: полный гард.** Обернуть hover-правила всех модулей в
  `@media (hover: hover)`. iPhone/десктоп не меняются; на Android исчезает
  sticky-hover артефакт. Полуавтоматическая обёртка скриптом + ручной проход
  исключений (правила, где hover дублирует :focus-visible/:active — проверить,
  что тап-feedback через :active сохраняется там, где он задуман).

### Замер скролла (в Блок 0)

Сценарий «флинг дневника 3 экрана вниз-вверх» на P30: записать в
`performance-reports/` % дропнутых кадров и longest frame через
`chrome://inspect` → Performance. Прогнать до/после A10, D1. Цель: ≥55 fps
медианно, ни одного кадра > 50 мс.
