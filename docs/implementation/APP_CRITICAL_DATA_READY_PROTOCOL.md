# Протокол реализации: стабильный первый кадр приложения

> **Дата:** 2026-07-19 **Статус:** завершено **Scope:** PIN boot, Phase A и
> первый render

## Критерий успеха

После loader приложение не показывает промежуточный локальный вопрос или
дефолтную шапку, которые исчезают после облачной синхронизации. Повторный mount
того же клиента использует уже подтверждённую Phase A без нового ожидания.

## Шаги и факты

1. **Диагностика — завершено.** Runtime-лог показал принудительный render по
   soft timeout 2,5 с после уже начавшейся синхронизации; persisted
   `last_sync_ts` также разрешал stale first frame без свежей Phase A.
2. **Контракт readiness — завершено.** Sync-runtime хранит готовность
   критических данных по `clientId`; при delta-старте сначала получает их одним
   batch-запросом, затем запускает остальной хвост синхронизации. Поздний ответ
   после выхода или переключения клиента не может опубликовать readiness.
3. **Первый кадр и fallback — завершено.** Старый порог 2,5 с заменён
   восьмисекундным аварийным окном только для деградировавшей сети. Boot-loader
   ждёт `heys:app-content-ready`, то есть commit реального React-экрана, а не
   только асинхронный вызов `root.render()`.
4. **PIN-переход — завершено.** LoginScreen сохраняет единое состояние
   «Проверяем PIN» до Phase A. Первая активация `anonymous → client` больше не
   запускает второй `AppAuthInit`; remount остаётся для logout и
   `client → client`.
5. **Регрессия — завершено.** Синтаксис прошёл; 47/47 тестов подтвердили
   readiness повторного mount, запрет stale render, порядок critical batch,
   барьер React commit и безопасный PIN-переход.
6. **Локальный runtime — завершено.** Scoped bundle собрал `boot-core`
   `d512e03efd36`, `boot-app` `f7dc84904c09` и `boot-init` `aa88e7072335`. На
   `localhost:3001` loader ушёл через 3,7 с уже поверх готовых шапки и основного
   блока; белого кадра не было, вопрос «Ел недавно или давно?» не появился ни
   сразу, ни через 3 секунды. Повторный мобильный smoke после PIN-правки показал
   готовые шапку и дневной экран без ошибок консоли; сам ввод PIN остаётся
   ручной проверкой владельца сессии, потому что тест не читает и не сохраняет
   учётные данные браузера.

## Риск

При реально недоступной сети критический sync ждёт до восьмисекундного fallback;
после него loader всё равно снимается только вместе с первым пригодным экраном,
а синхронизация продолжает работать в фоне.
