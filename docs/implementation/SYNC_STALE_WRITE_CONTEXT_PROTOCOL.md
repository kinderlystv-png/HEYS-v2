# Протокол реализации: восстановление sync после устаревшего write-context

## Проблема

Pending-запись сохраняла `_ctx` между браузерными сессиями. Uploader принимал
само наличие этого значения за достаточное условие и повторно отправлял уже
истёкший context. Сервер отклонял merge-save, запись оставалась первой в
критической очереди, а консоль заполнялась сообщениями о достигнутом лимите
повторов.

## Решение

Для context-required upload теперь всегда проверяется или выпускается
write-context текущей сессии. При сборке RPC-записи live-context имеет приоритет
над сохранённым `_ctx`; если актуальный context получить нельзя, данные остаются
в pending-очереди без небезопасной отправки или удаления.

## Проверка

- Синтаксис `heys_storage_supabase_v1.js` — корректен.
- `heys-storage-write-context-session.test.js` — 7/7 тестов пройдено.
- Регрессия подтверждает, что dayv2 с `ctx-stale` отправляется через merge-save
  с новым `ctx-live`.

## Дополнение: защита от retry-шторма

`scheduleClientPush` и immediate-upload теперь используют общий retry-бюджет.
После пяти неудачных RPC-попыток запись остаётся в durable pending-очереди, но
не планируется снова каждые 500 мс. Повторы используют exponential backoff, а
диагностический warning включает исходную ошибку merge-save.

Штатный server rejection `cross_client_*` или `invalid_profile_field` считается
терминально обработанным. Если fresh-row защита намеренно вернула `v: null`,
клиент не возвращает уже отклонённый payload в pending-очередь.
