# Протокол реализации: целостное завершение утреннего чек-ина

> **Дата:** 2026-07-19 **Статус:** завершено **Scope:** открытие после PIN,
> журнал шагов, финал и cloud-ack

## Критерий успеха

Чек-ин открывается только после полного первичного download, один раз проходит
зафиксированный план и не может объявить завершение, если после синхронизации
появилась обязательная проверка прошлых дней. Статус показывает последний
сохранённый шаг, а после опустошения очереди журнал получает явный cloud-ack.

## Шаги и факты

1. **Диагностика — завершено.** Полный лог показал открытие плана из события
   согласий ещё во время Phase A; прошлые дни приехали позже, поэтому
   `yesterdayVerify` был добавлен уже после первого `flow_complete`.
2. **Readiness-гейт — завершено.** Событие согласий не вычисляет чек-ин до
   полного `heysSyncCompleted` активного клиента; Phase A остаётся только
   подготовкой первого кадра.
3. **Финальный инвариант — завершено.** Перед session-флагом повторно сверяется
   текущее решение `YesterdayVerify`; новый обязательный шаг возвращает flow в
   `open` и блокирует ложный финал.
4. **Статус и cloud-ack — завершено.** Выбирается самый свежий локально
   сохранённый шаг; `heys:queue-drained` переводит pending-журнал в `synced`.
5. **Регрессия — завершено.** 32/32 точечных теста подтвердили readiness полного
   sync, повторную финальную проверку, восстановление журнала, актуальную
   подпись статуса и cloud-ack. Синтаксис трёх изменённых runtime-файлов прошёл.
6. **Локальный QA — завершено.** Scoped-сборка обновила `boot-app`
   (`e2ddf0133a6d`) и `postboot-3-ui-lazy` (`664cd660578e`); manifest/index
   согласованы. Web и API отвечают 200 на `localhost:3001` и `localhost:4001`.

## Риски

- Навигация пользователя назад может легитимно повторить сохранение сна; это не
  должно ошибочно считаться повторным запуском flow.
- Общая pending-очередь может не опустеть из-за несвязанной записи. В этом
  случае чек-ин остаётся честно помечен как сохранённый локально и не блокирует
  пользователя.
