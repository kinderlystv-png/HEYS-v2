# Протокол реализации: DayTab mobile stats fast path

> **Дата:** 2026-07-18 **Статус:** завершено **Scope:** только невидимая работа
> diary renderer на mobile stats/activity

## Критерий успеха

При переключении заполненного дня на мобильной вкладке «Отчёты»
`renderDiarySection()` должен вернуть insulin-индикатор до построения скрытых
diary-панелей. Видимый UI, данные дня и поведение вкладки «Питание» не меняются.

## Шаги и факты

1. **Baseline — завершено.** Изолированный профиль 390×844, demo day с 17
   позициями, CPU throttling 6×. Старая функция: commits
   `191 + 50 + 38.1 = 279.1 ms`, скрытая сборка до `5.3 ms`.
2. **Механизм — подтверждён.** `showDiary` проверялся после создания refeed,
   component factories и deferred-slot дерева, хотя mobile stats использует
   только insulin-индикатор.
3. **Исправление — завершено.** Проверка `!showDiary` перенесена сразу после
   создания insulin-индикатора.
4. **Регрессия — пройдена.** `day-diary-section-mobile-fast-path.test.js`
   подтверждает результат и отсутствие вызова скрытого refeed renderer.
5. **После — подтверждено.** Тот же сценарий: commits
   `182 + 73.6 + 0.5 = 256.1 ms`, скрытая сборка `0–0.1 ms`.
6. **Локальный runtime — готов.** Scoped `boot-day` и `boot-calc` собраны,
   свежий runtime доступен на `http://localhost:3001`.

## Остаточный риск

Первый DayTab commit остаётся `182–342 ms` при 6× CPU. Это наблюдение, а не
автоматическое разрешение на широкий refactor: задача возвращается в активную
очередь при реальном dead tap, INP выше 200 ms на целевом устройстве или
повторяемой потере действия.
