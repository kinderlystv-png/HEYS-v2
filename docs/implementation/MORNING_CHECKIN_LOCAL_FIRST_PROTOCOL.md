# Протокол реализации: local-first сохранение утреннего чек-ина

## Проблема

После локальной записи каждого шага мастер вызывал `flushPendingQueue(10000)` и
ждал очистки всей очереди. Если несвязанная cloud-запись оставалась pending,
кнопка «Далее» удерживалась до таймаута, хотя данные текущего шага уже были
безопасно сохранены локально.

## Решение

Успешная локальная запись сразу подтверждает шаг. Запись сохраняет статус
`saved_local` с `background_sync`, а штатный uploader продолжает отправку без
блокировки навигации. Ошибка самой локальной записи остаётся блокирующей.
Финальное завершение чек-ина следует тому же контракту.

## Проверка

При восстановлении мастер может смонтироваться раньше регистрации одного из
оставшихся шагов. `StepModal` слушает поздние регистрации и замораживает список
только после появления всех конфигураций, поэтому одиночный оставшийся шаг не
превращается в пустой экран.

- Синтаксис `heys_morning_checkin_v1.js` — корректен.
- `morning-checkin-stability.test.js` — 16/16 тестов пройдено.
- Контрактный тест запрещает возвращение синхронного ожидания
  `flushPendingQueue` в подтверждение шага и финал чек-ина.
