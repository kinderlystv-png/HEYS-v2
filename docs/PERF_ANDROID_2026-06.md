# Производительность на Android (Huawei P30): причины и план правок

Дата: 2026-06-13. Контекст: на iPhone 13 приложение плавное, на Huawei P30 (PWA
из Chrome) — подтормаживания и фризы «везде понемногу». Все причины ниже
верифицированы по коду с file:line.

> **Прогон №2 + релиз-безопасный план:**
> [PERF_ANDROID_2026-06_PASS2.md](PERF_ANDROID_2026-06_PASS2.md). По итогам
> верификации пункты 1.2 (отключение blur/анимаций) и 1.4 (passive-листенеры)
> этапа 1 **отменены** — см. там раздел «Поправки к прогону №1».

## 1. Почему iPhone 13 прощает, а P30 — нет

- **CPU.** A15 Bionic (2021) в ~2.5–3× быстрее Kirin 980 (2019) в single-core, а
  JS однопоточен. Работа, укладывающаяся на iPhone в бюджет кадра 16.7 мс, на
  P30 занимает 40–50 мс → пропуск кадров.
- **GPU.** `backdrop-filter: blur()` на Mali-G76 в Chrome стоит в разы дороже,
  чем на Apple GPU в Safari (у WebKit оптимизированный путь для блюра).
- **Style recalc.** Стоимость пересчёта стилей растёт с объёмом CSS (у нас ~2.4
  МБ активных правил) и числом DOM-узлов — налог на каждую мутацию DOM, который
  слабый CPU не скрывает.
- **EMUI.** PowerGenie и термотроттлинг дополнительно режут частоты.

Вывод: приложение живёт «на грани бюджета кадра». Чинить — значит снижать
main-thread и GPU-нагрузку; запас по железу появится сам.

Детали по №1: interceptor на каждый dayv2-write делает decompress/`JSON.parse`
всего дня для day-trace (`:4469–4479`), затем **второй** `tryParse(v)` для
stamping (`:4564`), плюс для products-ключей overlay-guard с `getItem` + parse
массива (`:4526–4561`), затем re-stringify. Каждый ввод граммов = несколько
полных сериализаций дня на main thread.

Вне кода (смягчение, не решение): на P30 обновить Chrome/WebView, отключить
оптимизацию батареи для Chrome, включить производительный режим EMUI.

## 3. План внедрения по степени эффективности

### Этап 1 — perf-lite режим + quick wins (≈1 день, низкий риск)

Ожидание: снимает 50–70% жалоб, т.к. бьёт по причинам №2–№5.

- **1.1. Флаг `HEYS.perfLite`.** Android + (`navigator.deviceMemory ≤ 6` или
  короткий бенч-пробник на старте) → `<html class="perf-lite">`. Переключатель в
  настройках, дефолт для слабых устройств — включён.
- **1.2. CSS в perf-lite:** глобально отключить `backdrop-filter` (заменить на
  rgba/непрозрачный фон) и декоративные `infinite`-анимации (shimmer, pulse,
  idle-эффекты). Плюс уважать `prefers-reduced-motion`.
- **1.3. Логирование:** `console.info` нормального пути в interceptor (`:4512`)
  — за debug-флаг; в log-trace на Android — sampling (~10%) и обрезка больших
  объектов до N символов перед сериализацией.
- **1.4. `passive: true`** везде, где нет `preventDefault` (аудит списка из
  п.2.5). Где `preventDefault` нужен (drag, edit-grams) — вешать листенер на
  элемент **на время жеста**, не на `document` постоянно.
- **1.5. `content-visibility: auto` + `contain-intrinsic-size`** на тяжёлые
  оффскрин-секции: карточки приёмов в дневнике, строки длинных списков.

### Этап 2 — interceptor и transitions (2–3 дня, средний риск)

- **2.1. Один parse на write.** Объединить trace-блок (`:4469`) и stamp-блок
  (`:4564`): парсить/декомпрессить `v` один раз, переиспользовать результат.
  **Не менять семантику** stale-writer detection, overlay self-heal guard и
  stamping (инварианты №5/№7 CLAUDE.md) — только убрать дублирование. Тесты в
  `TESTS/` до и после.
- **2.2. `transition: all` → явные свойства**
  (`transform, opacity, background-color, border-color, box-shadow`). 364 места
  — полуавтоматическая замена скриптом + ручной проход исключений.
- **2.3. Таймеры:** тики 1000 мс — только при `!document.hidden` (частично уже
  так); reconcile/tick — через `requestIdleCallback` с fallback.

### Этап 3 — структурные (неделя+, делать по результатам замеров)

- **3.1. CSS-диета.** Тяжёлые модули — в lazy-загрузку по вкладкам по образцу
  `main-deferred.css`: кандидаты `fingers.css` (268К, грузить при входе в
  fingers), `600-steps-and-aps.css` (116К) и далее по списку. PurgeCSS-отчёт по
  мёртвым правилам.
- **3.2. Виртуализация** длинных списков (дневник, продукты): простой windowing
  по viewport без новых зависимостей.
- **3.3. Boot-диета.** Разбор boot-core/boot-app по `bundle-analysis/`,
  выселение редко используемого в postboot.
- **3.4. (Опционально, большой риск)** Перевод локального зеркала с синхронного
  localStorage на async-очередь — только если после этапов 1–2 замеры всё ещё
  показывают спайки на write.

## 4. Как мерить (до/после каждого этапа)

- **P30 по USB:** `chrome://inspect` → Performance-запись сценариев «добавить
  продукт», «скролл дневника», «переключить вкладку». Смотреть INP и long tasks
  (>50 мс).
- **Прокси без устройства:** Lighthouse mobile, CPU throttle 4–6×.
- **Бюджет:** INP < 200 мс на P30; ни одного long task > 200 мс в сценарии
  «добавить продукт».
- **С поля:** в log-trace уже есть транспорт батчей — добавить отправку
  web-vitals (INP/LCP/CLS), чтобы видеть реальные устройства клиентов.

## 5. Риски и инварианты

- **Interceptor (2.1)** — зона инвариантов: stale-writer detection, overlay
  self-heal guard, stamping. Менять структуру (single parse), не логику;
  обязательны тесты.
- **CSS no-touch зоны** (шапка `styles/main.css`): `@keyframes`, `.confetti-*`,
  `.water-ring`/`.water-splash`, safe-area, `.mpc-*`. Perf-lite отключает
  эффекты классом поверх, не редактирует эти блоки.
- **Свайпы/pull-refresh:** `passive: false` там осознанный — менять только после
  проверки жестов на реальном Android.
