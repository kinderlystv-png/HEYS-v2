# Mobile Web App Audit (apps/web)

## 🚫 Disabled Components

- **Telegram Mini App (`apps/tg-mini`)**: Disabled in workspace and scripts.
- **Expo Mobile App (`apps/mobile`)**: Disabled in scripts.

## 📱 Current State: Legacy v12 (PRODUCTION)

Приложение (`apps/web`) работает в **Legacy-режиме** — и это **рабочая
production версия**.

### 1. Entry Point (`index.html`)

- **Structure**: Файл `index.html` — это основная точка входа (1170 строк).
- **Loading**: Загружает JS-файлы через `<script>` теги (`heys_core_v12.js`,
  `heys_day_v12.js`, `heys_storage_supabase_v1.js` и др.).
- **Rendering**: Использует `ReactDOM.createRoot` внутри inline `<script>` для
  рендера компонента `App`.
- **State Management**: Глобальные переменные (`window.HEYS.*`), localStorage
  через утилиты `U.lsGet/lsSet`.

### 2. Legacy JS Modules (PRODUCTION CODE)

Эти файлы — **работающая бизнес-логика**:

- `heys_core_v12.js` — поиск продуктов, парсинг, управление рационом (869 строк)
- `heys_day_v12.js` — статистика дня, приёмы пищи, тренировки (1112 строк)
- `heys_user_v12.js` — профиль пользователя, HR-зоны
- `heys_reports_v12.js` — отчётность за период
- `heys_storage_supabase_v1.js` — синхронизация с Supabase (494 строки)
- `heys_storage_layer_v1.js` — кэширование и localStorage (164 строки)
- `heys_simple_analytics.js` — минимальная аналитика (217 строк)

### 3. Modern Codebase (`src/`) - **НЕ ИСПОЛЬЗУЕТСЯ**

- **Статус**: В папке `apps/web/src/` есть заглушки (`App.tsx`, `main.tsx`), но
  они **не подключены** к `index.html`.
- **Следствие**: Код в `src/` — это **мёртвый код**, не влияет на приложение.

### 4. Глобальные зависимости (КРИТИЧНО для миграции)

**Все legacy модули завязаны на глобальные переменные:**

- `window.HEYS.currentClientId` — используется в 15+ местах для определения
  активного клиента
- `window.HEYS.saveClientKey()` — вызывается из `lsSet()` для синхронизации с
  Supabase
- `window.HEYS.utils.lsGet/lsSet` — обёртки над localStorage с автоматическим
  namespace
- `window.HEYS.cloud.*` — методы для работы с Supabase
- `window.HEYS.analytics.*` — трекинг производительности

**⚠️ При любой миграции эти глобальные объекты ДОЛЖНЫ сохраниться**, иначе
сломается:

- Синхронизация с облаком
- Переключение клиентов
- Бэкап система
- Аналитика

### 5. Стили и Mobile UI

- **CSS структура**: `styles/critical.css` (inline), `styles/main.css` (1046
  строк).
- **CSS переменные**: Используются для темизации (`--bg`, `--card`, `--text`,
  `--acc`).
- **Responsive**: Базовый `viewport` meta tag, но нет продвинутых mobile
  паттернов.
- **Touch targets**: Минимальные, но работают.

### 6. Критическая оценка качества

#### ✅ Что работает хорошо:

- **Production-ready**: Приложение стабильно работает, данные сохраняются.
- **Supabase интеграция**: Работает авторизация, синхронизация клиентов, RLS.
- **Offline-first**: Fallback на localStorage, если нет подключения.
- **Performance hints**: Ручные `preload`/`prefetch` для оптимизации.
- **Backup система**: Встроенный механизм бэкапов продуктов и данных дня.

#### ⚠️ Проблемы и риски:

- **Maintainability**: Логика размазана по `index.html` (1170 строк) и
  глобальным JS.
- **No TypeScript**: Нет type safety, легко допустить ошибки.
- **No modern tooling**: Нет автоматического tree-shaking, code splitting, HMR
  для логики.
- **Coupling**: Компоненты завязаны на глобальные объекты (`window.HEYS.*`).
- **Mobile UX**: Базовый responsive, но нет сложных мобильных паттернов
  (pull-to-refresh, swipe navigation).

## 🚀 Рекомендации для "качественной мобильной веб-версии"

### ⚠️ КРИТИЧНО: НЕ ЛОМАТЬ СУЩЕСТВУЮЩЕЕ

**Legacy v12 — это production.** Любая миграция должна быть **постепенной** и
**безопасной**.

### Option A: Постепенная миграция (Рекомендуется)

**Цель**: Переход на `src/` структуру **БЕЗ** потери работающей
функциональности.

**План действий:**

1. **Фаза 1: Подготовка (безопасная)**
   - Создать **параллельную** entry point в `index-modern.html` (для
     тестирования).
   - Портировать `App` компонент из `index.html` в `src/App.tsx`.
   - **КРИТИЧНО**: Сохранить все глобальные объекты `window.HEYS.*` для
     совместимости.
   - Оставить `index.html` нетронутым как fallback.

2. **Фаза 2: Изоляция модулей**
   - Обернуть legacy JS (`heys_core_v12.js`, `heys_day_v12.js`) в ES модули.
   - Экспортировать их как React components/hooks из `src/legacy/`.
   - **ВАЖНО**: Не менять внутреннюю логику, только обёртки.
   - Постепенно рефакторить внутреннюю логику (после тестирования).

3. **Фаза 3: Переключение**
   - Когда `index-modern.html` будет полностью протестирован — сделать его
     основным.

- Старый `index.html` хранить через git history (корневой `archive/` удалён).
- **Обязательно**: Проверить работу Supabase sync, переключение клиентов,
  бэкапы.

**Преимущества:**

- ✅ Сохраняется работающая версия на каждом этапе.
- ✅ Можно откатиться в любой момент.
- ✅ TypeScript, HMR, modern tooling постепенно.

**Риски:**

- ⚠️ Дублирование кода на время миграции.
- ⚠️ Может занять несколько итераций.

### Option B: Улучшение Legacy (Минимальный риск)

**Цель**: Навести порядок в существующей структуре **без** перехода на `src/`.

**План действий:**

1. **Рефакторинг `index.html`**
   - Вынести inline `<script>` в отдельный файл `heys_app_v12.js`.
   - Разбить на логические блоки (DayTabWithCloudSync, RationTabWithCloudSync и
     т.д.).
   - **НЕ ТРОГАТЬ**: Глобальные объекты `window.HEYS.*` остаются как есть.

2. **Улучшение legacy модулей**
   - Добавить JSDoc комментарии для type hints (без изменения логики).
   - Улучшить читаемость кода (форматирование, комментарии).
   - Добавить error boundaries в React компонентах.

3. **Mobile UX улучшения (ФОКУС для этой опции)**
   - **Touch gestures**: Swipe для переключения вкладок.
   - **Pull-to-refresh**: Для синхронизации с облаком.
   - **Responsive**: Улучшенная адаптивность для разных экранов.
   - **Performance**: Оптимизация рендеринга списков (виртуализация).
   - **Offline UX**: Индикаторы состояния синхронизации.

**Преимущества:**

- ✅ Минимальный риск сломать production.
- ✅ Быстрее, чем полная миграция.
- ✅ Можно делать постепенно.

**Недостатки:**

- ❌ Не получаем TypeScript и modern tooling.
- ❌ Сохраняется техдолг.
- ❌ Потолок по качеству кода.

### Option C: Гибридный подход (Баланс)

**Цель**: Оставить legacy JS, но обернуть их в modern build pipeline.

**План действий:**

1. Переключить `index.html` на `<script type="module" src="/src/main.tsx">`.
2. В `src/main.tsx` **импортировать** legacy JS как side-effects:
   ```typescript
   // src/main.tsx
   import './legacy-loader.ts'; // Загружает все heys_*.js
   import { App } from './App';
   ```
3. Создать `src/legacy-loader.ts` для инициализации глобальных объектов.
4. Постепенно рефакторить отдельные модули в TypeScript (не трогая работающие).

**Преимущества:**

- ✅ Получаем Vite HMR и bundling.
- ✅ Можно постепенно мигрировать модули.
- ✅ Legacy JS продолжает работать через глобальные объекты.
- ✅ TypeScript для нового кода.

**Риски:**

- ⚠️ Нужна аккуратная интеграция глобальных переменных (но это решаемо).
- ⚠️ Возможны конфликты ESM/UMD (legacy модули ожидают `window.React`).
