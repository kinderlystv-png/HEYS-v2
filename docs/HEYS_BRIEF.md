# ЧАСТЬ I: БИЗНЕС-ЯДРО

## 1. Суть HEYS (elevator pitch)

> **HEYS — не трекер. HEYS — экосистема.**

**Экосистема = приложение + живой куратор + протоколы сопровождения.**

| Что продаём                     | Не фичи                   |
| ------------------------------- | ------------------------- |
| Сопровождение и удержание курса | Счётчик калорий           |
| Человека, которому не всё равно | Автоматизированные отчёты |
| Внешний контур ответственности  | DIY-трекинг               |

**Центральный стержень:**

- Есть человек, которому **не всё равно** на жизнь пользователя
- Куратор **вовлечён в контекст жизни** и удерживает фокус на здоровье

---

## 2. Целевой клиент

| Параметр              | Значение                                                                         |
| --------------------- | -------------------------------------------------------------------------------- |
| **Аудитория**         | Люди, которые хотят худеть/следить за здоровьем, но **в одиночку не получается** |
| **Ключевая боль**     | «Я знаю, что контроль работает, но я не выдерживаю рутину и срываюсь»            |
| **Почему платят**     | Нужен **внешний контур ответственности** (как тренер/диетолог/марафон)           |
| **Ключевая ценность** | Куратор рядом (поддержка при срыве) + снимает рутину (ведёт дневник)             |

## 12. Онбординг (7 дней Pro-триала)

### 12.1 Флоу онбординга клиентов (Trial Machine v3.0)

```
[Лендинг heyslab.ru] → заявка
         ↓
   [leads таблица] ← heys-api-leads (cloud function)
         ↓
[Админка куратора] → секция «Лиды с сайта» 🌐
         ↓ admin_get_leads(status='new')
  [Список лидов]
         ↓ Куратор связывается с клиентом
         ↓ Обсуждает программу, выдаёт PIN
         ↓
[Кнопка «Создать клиента»] → диалог ввода PIN
         ↓ admin_convert_lead(lead_id, pin)
   [Клиент создан]
   status='none'
   trial_queue: 'queued'
         ↓
[Кнопка «Активировать trial»] → диалог выбора даты
         ↓ admin_activate_trial(client_id, start_date)
         ├─ Если start_date = сегодня → trial (7 дней)
         └─ Если start_date > сегодня → trial_pending
                  ↓ (дата наступает)
              [trial] 7 дней
                  ↓ (истекает)
             [read_only]
                  ↓
             [Paywall]
```

### 12.2 Ключевые изменения v3.0

- **Нет автостарта триала** — куратор выбирает момент запуска
- **trial_pending статус** — одобрен, но дата старта в будущем (например, клиент
  уезжает через неделю)
- **admin_get_leads** — лиды с лендинга видны в админке
- **admin_convert_lead** — конвертация лида → клиент (с PIN)
- **admin_activate_trial** — активация с выбором даты

## 14. Границы ответственности

### 14.1 Куратор делает

- Ведёт дневник питания (расшифровка фото, внесение в базу)

- ❌ Не интерпретирует анализы как диагноз

| Ситуация                                   | Действие                                                   |
| ------------------------------------------ | ---------------------------------------------------------- |

| **Беременность/ГВ**                        | Нет дефицита, мягкий режим + консультация врача            |
| **Подозрение на РПП**                      | Бережно рекомендовать специалиста; при отказе — прекращаем |

## 18.K Фискализация, чеки и ОФД (54-ФЗ) `[BIZ]` `[LEGAL]` `[BLOCKER]`

### 18.K.1 Почему это обязательно

- При приёме оплаты от **физлиц** за товары/работы/услуги организация/ИП, как
  правило, обязаны применять **ККТ** и формировать **кассовый чек** (в т.ч. при
  онлайн-оплате).
- “Подтверждение оплаты” от платёжного сервиса **не заменяет** фискальный чек:
  чек должен быть фискализирован и передан через ОФД в ФНС.

### 18.K.2 Что такое ОФД и как устроен поток

**Фискализация** = регистрация операции оплаты через ККТ: ККТ формирует чек →
данные уходят **оператору фискальных данных (ОФД)** → затем в ФНС.

### 18.K.3 Что именно нужно обеспечить в HEYS (Definition of Done)

- После успешной оплаты автоматически:
  1. формируется чек (приход) с корректными признаками,
  2. чек уходит через ОФД в ФНС,
  3. электронный чек уходит клиенту на **email/телефон**,
  4. возврат/частичный возврат формирует чек возврата.

---

### 18.K.4 Варианты реализации (выбрать один)

#### Вариант A — Своя/облачная онлайн-касса + ОФД (через ЮKassa)

Схема:

- ты подключаешь онлайн-кассу (часто “облачную”) и ОФД,
- в личном кабинете ЮKassa включаешь отправку чеков и подключаешь
  кассу/провайдера, ЮKassa передаёт данные для фискализации.

Плюсы:

- максимальная гибкость (номенклатура, возвраты, сценарии).

Минусы:

- больше договоров/настроек (касса, ОФД, иногда фискальный накопитель у
  провайдера).

#### Вариант B — «Чеки от ЮKassa» (облачная касса через партнёра)

ЮKassa предлагает режим, когда **облачную кассу партнёра** подключаешь через
личный кабинет ЮKassa: "Чеки от ЮKassa" — это SaaS-решение, которое закрывает
54-ФЗ за тебя (касса есть, просто арендуется).

Плюсы:

- проще и быстрее старт (минимум инфраструктуры, касса через партнёра).

Минусы/проверки:

- подходит не для всех сценариев (нужно сверить ограничения “обычных платежей” и
  вашу модель).

**Рекомендация для HEYS (быстрый старт):** начать с **Варианта B**, если он
совместим с вашим сценарием (подписка без автосписаний, услуги, возвраты). Если
ограничения мешают — Вариант A.

---

### 18.K.5 Что должно быть в чеке (что согласовать с бухгалтером)

#### 1) Номенклатура (название позиции)

Пример (как “услуга”):

- `HEYS Base — доступ к сервису и сопровождение (1 месяц)` Для Pro/Pro+
  аналогично.

#### 2) “Предмет расчёта” и “способ расчёта”

Ключевой выбор, который влияет на интеграцию:

- **Проще всего**: чек как **полный расчёт** в момент оплаты за месяц (подходит,
  если вы реально оказываете услугу/предоставляете доступ на оплаченный период).
- Сложнее: “предоплата/аванс” + “зачёт предоплаты” (обычно нужно, если бухгалтер
  так требует под вашу модель).

#### 3) НДС в чеке

На спецрежимах часто “без НДС”, но это подтверждает бухгалтер (зависит от режима
ИП/УСН/др.).

---

### 18.K.6 Отправка электронного чека клиенту

- Электронный чек отправляется при наличии email/телефона (через ОФД/кассу).
- **Сроки:** Фиксируем по схеме, согласованной с бухгалтером и кассой. Для
  онлайн-услуг стандартная практика — формировать чек **в момент подтверждения
  оплаты**. Точные требования зависят от сценария расчёта (полный расчёт / аванс
  / зачёт).

**Практика для HEYS:** на экране оплаты собрать email или телефон для чека и
хранить в профиле.

| `payment.canceled`            | Платёж отменён               | ❌ НЕ активировать               |
| `payment.waiting_for_capture` | Двухстадийный (если включён) | ⏳ Ждать capture                 |
| `refund.succeeded`            | Возврат завершён             | 🔄 Откатить доступ (по правилам) |

**Формат уведомления:**

```json
{
  "type": "notification",
  "event": "payment.succeeded",
  "object": { "id": "...", "status": "succeeded", "amount": {...}, ... }
}
```

**Критичные правила:**

1. **HTTP 200 обязателен** — иначе ЮKassa ретраит до 24ч
   ([docs](https://yookassa.ru/developers/using-api/webhooks))
2. **Проверка подлинности:**
   - Allowlist IP-адресов ЮKassa
   - GET статуса объекта через API перед активацией
3. **Идемпотентность:** дедуп по `(event, object.id)` — webhook может прийти
   повторно

#### 18.L.2 Идемпотентность

| Где                    | Как                                                                                                                    |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **Создание платежа**   | Заголовок `Idempotence-Key` (UUID), актуален 24ч ([docs](https://yookassa.ru/developers/using-api/interaction-format)) |
| **Обработка webhook**  | Дедуп по `dedupe_key = event + ":" + object.id`                                                                        |
| **Активация подписки** | Проверка: уже `active`? → skip                                                                                         |

#### 18.L.3 Схема таблиц PostgreSQL

**`payments`** — наши платежи:

| Поле                  | Тип         | Описание                                    |
| --------------------- | ----------- | ------------------------------------------- |
| `id`                  | UUID        | Внутренний ID                               |
| `yookassa_payment_id` | TEXT        | ID платежа в ЮKassa                         |
| `client_id`           | UUID        | FK → clients                                |
| `plan_code`           | TEXT        | `Base` / `Pro` / `Pro+`                     |
| `amount_value`        | NUMERIC     | Сумма                                       |
| `amount_currency`     | TEXT        | `RUB`                                       |
| `status`              | TEXT        | `created`/`succeeded`/`canceled`/`refunded` |
| `idempotence_key`     | TEXT        | Для защиты от дублей                        |
| `confirmation_url`    | TEXT        | URL оплаты (redirect)                       |
| `is_test`             | BOOLEAN     | Тестовый платёж?                            |
| `created_at`          | TIMESTAMPTZ | —                                           |

**`yookassa_webhook_events`** — лог webhook (обязательно для дебага):

| Поле                | Тип         | Описание                   |
| ------------------- | ----------- | -------------------------- |
| `id`                | SERIAL      | —                          |
| `event`             | TEXT        | `payment.succeeded` и др.  |
| `object_id`         | TEXT        | ID объекта в ЮKassa        |
| `dedupe_key`        | TEXT        | `event:object_id` (UNIQUE) |
| `received_at`       | TIMESTAMPTZ | Когда получили             |
| `processed_at`      | TIMESTAMPTZ | Когда обработали           |
| `processing_result` | TEXT        | `ok` / `error:reason`      |
| `raw_body`          | JSONB       | Полный payload             |

#### 18.L.4 Статусы и переходы

**Статусы платежа (внутренние):**

```
created → succeeded → (refunded)
       ↘ canceled
```

**Переходы подписки:**

```
trial → read_only (истёк триал, нет оплаты)
read_only → active (только после payment.succeeded!)
active → read_only (refund / истёк период)
```

#### 18.L.5 Риски интеграции

| Риск                          | Последствие                        | Митигация                            |
| ----------------------------- | ---------------------------------- | ------------------------------------ |
| Webhook не отдал 200          | Ретраи 24ч, дубли активаций        | Idempotency + логирование            |
| Нет проверки подлинности      | Фейковые webhook от злоумышленника | IP allowlist + GET статуса по API    |
| Нет дедупа webhook            | Двойная активация/возврат          | UNIQUE constraint на dedupe_key      |
| СНО в чеке ≠ режим выручки    | Налоговые проблемы                 | Настроить кассу под правильный режим |
| Нет email/телефона покупателя | Проблемы с электронным чеком       | Собирать в форме оплаты/профиле      |

#### 18.L.6 План запуска оплаты

1. ⏸️ **Бухгалтер:** УСН/ОСНО, ОКВЭД, модель продажи, параметры чека
2. ⏸️ **ЮKassa:** регистрация → `shopId`
3. ⏸️ **54-ФЗ:** выбор варианта (A/B), касса + ОФД, тестовый чек
4. 🔧 **Webhook** `payment_succeeded` → активация подписки (~2-4 часа)
5. 🔧 **UI:** кнопка "Оплатить" + редирект на ЮKassa (~2 часа)
