```
- Сведения о самочувствии и общем состоянии
```

**Технические данные:**

```
- IP-адрес устройства пользователя
- Идентификатор браузера (User-Agent)
- Сведения об устройстве, операционной системе
- Дата и время использования сервиса
- Действия пользователя в приложении (логи)
```

## 8. Меры по обеспечению безопасности ПДн

```
Применяются правовые, организационные и технические меры в соответствии
со ст. 18.1 и ст. 19 152-ФЗ:

Правовые:
- Ответственным за организацию обработки ПДн является сам оператор (ИП)
  согласно Положению об обработке персональных данных (утв. ИП Поплавский А. С.,
  22.05.2026)
- Утверждены локальные документы по обработке ПДн (Положение об обработке)
- Получаются согласия субъектов на обработку в форме простой электронной
  подписи: проставление отметки в чекбоксе после ознакомления с документом,
  с фиксацией IP-адреса, User-Agent, даты/времени, версии документа в
  аудит-логе (data_access_audit_log). Для специальной категории — данных о
  состоянии здоровья — оформляется отдельное согласие в той же форме.

Организационные:
- Контроль доступа: только сам ИП имеет доступ к данным клиентов;
  привлечённые на основании договора кураторы (при наличии) — в объёме,
  необходимом для оказания услуг
- Логирование действий с ПДн в системе (data_access_audit_log)
- Ведётся учёт запросов субъектов ПДн (data_subject_requests) для
  соблюдения 10-рабочедневного срока (152-ФЗ ст. 21 ч. 4)

Технические:
- Шифрование передачи данных (HTTPS / TLS 1.3, HSTS)
- Шифрование хранилища (Yandex Cloud Postgres encryption-at-rest +
  column-level AES-256 для health-data через HEYS_ENCRYPTION_KEY)
- Хеширование PIN-кодов посредством bcrypt с автоматически генерируемой
  солью (cost factor 12)
- Ограничение попыток ввода PIN (rate limiting + блокировка после
  10 неудачных попыток за 15 минут)
- Управление секретами через Yandex Lockbox (PG_PASSWORD, JWT_SECRET,
  HEYS_ENCRYPTION_KEY, S3-ключи, VAPID_PRIVATE_KEY)
- Резервное копирование с шифрованием (Yandex Object Storage с SSE)
- Веб-уровень: Content Security Policy, X-Frame-Options, X-Content-Type-Options,
  HttpOnly + Secure + SameSite cookies для session-токенов
- Мониторинг событий безопасности (security_events + автоматические
  Telegram-алерты при подозрительной активности)
```


> **Внутренний комментарий (не для подачи в РКН — для аудита оператора):**
>
> Сервис использует Telegram-канал для системных уведомлений клиентам (drip о
> статусе подписки). Содержание этих уведомлений было намеренно
> **минимизировано** (правка `heys-cron-trial-drip` от 2026-05-20) — имена,
> телефоны и любые ПДн в outbound-трафик к Telegram API не передаются. В
> сообщениях фигурируют только: тип события
> (welcome/mid/prepay/lastcall/expired) и количество дней до конца триала (целое
> число, не ПДн).
>
> Telegram `chat_id` (идентификатор чата) технически передаётся, но он:
>
> 1. присваивается самим Telegram'ом при создании чата пользователя с ботом;
> 2. известен только при том условии, что пользователь сам инициировал разговор
>    с ботом (через PIN deep-link);
> 3. не позволяет идентифицировать субъекта ПДн без доступа к Telegram-системе.
>
> Поэтому квалификация «передачи персональных данных» к outbound-трафику бота не
> применима. Декларация выше — правдива.
>
> Если в будущем будет принято решение слать клиентам персонализированные
> сообщения с именем/деталями — потребуется подать **отдельное** уведомление о
> трансграничной передаче и взять у пользователя соответствующее согласие (ст.
> 12 152-ФЗ).

> **Push-уведомления (VAPID):** payload зашифрован end-to-end ключами клиента,
> push-серверы (Mozilla / Google / Apple) маршрутизируют зашифрованные пакеты,
> не имея ключа для расшифровки. Не квалифицируется как трансграничная передача
> ПДн.
