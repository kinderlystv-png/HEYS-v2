# Политика обращения с архивными снимками legal-документов

## Принцип frozen snapshots

Архивные снимки (`apps/web/public/docs/v1.2/`, `v1.5/`, и т. п.) — это
**доказательство акцепта**: каждая версия содержит ровно тот текст, на который
клиент согласился, нажимая чекбокс «Принимаю Пользовательское соглашение /
Политику конфиденциальности / Согласие на обработку данных о здоровье».

Любое изменение **условий услуги** в архивной версии = нарушение принципа
доказуемости акцепта и потенциальный юридический риск при споре с клиентом или
проверке РКН.

## Что считается «предметом акцепта» (трогать запрещено)

| Категория                                   | Примеры                                             |
| ------------------------------------------- | --------------------------------------------------- |
| **Тарифы и порядок оплаты**                 | «Pro 1490 ₽/мес», «возврат за неиспользованные дни» |
| **Права и обязанности сторон**              | «Клиент обязуется», «Оператор предоставляет»        |
| **Сроки**                                   | «Триал 7 дней», «возврат в течение 14 дней»         |
| **Перечень обрабатываемых данных**          | «Имя, телефон, дневник питания…»                    |
| **Категории получателей ПДн**               | «Куратор, ЮKassa, Yandex Cloud»                     |
| **Цели обработки ПДн**                      | «Предоставление услуг, аналитика, маркетинг»        |
| **Сроки хранения ПДн**                      | «До удаления аккаунта + 30 дней»                    |
| **Меры защиты ПДн**                         | «Шифрование, контроль доступа, логирование»         |
| **Дисклеймеры медицинской ответственности** | «Не является медицинской услугой»                   |

## Что НЕ считается «предметом акцепта» (правка допускается)

| Категория                                                                                                        | Обоснование                                                                                                                                                                                                                                           |
| ---------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Контактные данные** оператора в разделе «Контакты» (Telegram-handle, email, телефон поддержки, почтовый адрес) | Это **идентификация оператора**, а не условие услуги. Аналогия: компания переехала в новый офис — старый адрес перестаёт работать. Сохранять его в архиве ради формальной неизменности = причинять реальный вред клиенту, который не сможет связаться |
| **Опечатки** в любом разделе                                                                                     | Орфография не меняет условий                                                                                                                                                                                                                          |
| **Гиперссылки на внешние сайты** если они переехали                                                              | URL — это техническая идентификация, не условие                                                                                                                                                                                                       |

## Конкретное решение 2026-05-20: правка @heys_support → @heyslab_support

**Что произошло:** в архиве `v1.2/user-agreement.md:202` был указан
Telegram-handle поддержки `@heys_support`. На повторной проверке выяснилось, что
этот handle **не принадлежит оператору** (ИП Поплавский имеет `@heyslab_support`
как основной канал; `@heys_support` либо чужой, либо несуществующий).

**Что сделано:** заменено на `@heyslab_support` во всех 4 местах:

- [apps/web/public/docs/user-agreement.md:244](../../../apps/web/public/docs/user-agreement.md)
  (текущий)
- [apps/web/public/docs/v1.5/user-agreement.md:244](../../../apps/web/public/docs/v1.5/user-agreement.md)
- [apps/web/public/docs/v1.2/user-agreement.md:202](../../../apps/web/public/docs/v1.2/user-agreement.md)
- [docs/legal/user-agreement.md:244](../user-agreement.md)

**Обоснование правки в архивных снимках (v1.2, v1.5):**

1. Раздел «Контакты» — справочная информация, не предмет акцепта.
2. Старый handle указывал в **никуда** (или к чужому профилю), что вредит
   клиенту: он не может связаться с поддержкой.
3. Бамп версии не требуется — это **typo-fix контактной идентификации**, а не
   изменение условий услуги.
4. Юридическая практика: контакты оператора обновляются автоматически без нового
   акцепта.

**Что НЕ менялось:** ни одно условие услуги (тарифы, сроки, права, цели
обработки, меры защиты, сроки хранения) не было затронуто. Подписанный клиентом
контракт остаётся в силе по своим условиям.

## Правило для будущих агентов / разработчиков

Перед любой правкой архивного снимка ответь на вопрос:

> «Если клиент пришёл бы в суд с распечаткой архивной версии, на которую
> подписал согласие — изменилось ли бы решение суда из-за моей правки?»

- **Да, изменилось бы** → не трогать. Создать новую версию с бампом, новый
  re-consent flow.
- **Нет, не изменилось бы** (контактные данные, опечатки, обновление URL) →
  можно править без бампа, зафиксировать в этом документе.
