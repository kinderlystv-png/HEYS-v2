# Питание, расчёты дня и инсулиновая волна

> **Статус:** ключевые source-контракты проверены 2026-07-18 **Охват:** суммы
> приёмов и дня, нормы, источник нутриентов, расчёт и показ инсулиновой волны,
> fallback при поздней загрузке **Не подтверждено:** медицинская валидность
> коэффициентов, production-конфиг, браузерное поведение и полнота всех
> аналитических потребителей

## Граница системы

Контур состоит из двух разных по надёжности уровней:

```text
product / meal-item snapshot
          ↓
HEYS.models.mealTotals → HEYS.dayCalculations.calculateDayTotals
          ↓                         ↓
  дневные БЖУ/ккал          UI, scoring, advice, storage-derived totals

meal time + nutrients + profile + day context
          ↓
HEYS.InsulinWave.calculate
          ↓
эвристическая оценка длительности/формы волны и UI-статус
```

Первый уровень агрегирует сохранённые пищевые данные. Второй строит модельное
объяснение по множеству коэффициентов. Инсулиновая волна не измеряет глюкозу или
инсулин и не должна трактоваться как медицинский показатель.

## Суммы приёма и дня

`mealTotals` разрешает каждую позицию через модель продукта, нормализует поля на
100 г и масштабирует их на `item.grams`. Ключ кеша включает id приёма, snapshot
его позиций и сигнатуру product index. Это позволяет обновлять итог при
изменении stamp или состава каталога.

`calculateDayTotals` складывает результаты `mealTotals`, округляет основные
показатели до целого и отдельно считает средневзвешенные GI/harm по граммам.
Следствие: GI дня здесь имеет другую семантику, чем GI внутри инсулиновой волны,
где он взвешивается по углеводам. Называть эти числа взаимозаменяемыми нельзя.

Нормы строятся из целевого калоража и процентов. В текущем коде белок
переводится через net Atwater factor `3`, углеводы через `4`, жир через `9`;
fallback этих значений зашит прямо в расчёт. Это продуктовый контракт HEYS, а не
универсальная таблица энергетической ценности.

## Источник нутриентов

Канонический путь суммы приёма — `HEYS.models.getProductFromItem`. Модель
продукта умеет учитывать item snapshot, поэтому исторический день не должен
полностью зависеть от текущего каталога. Инсулиновая волна отдельно реализует
fallback `product → item snapshot → default` для части полей.

У волны есть важные defaults: отсутствующий GI становится `50`, а отсутствующие
нутриенты — нулями. Масса позиции проходит через
`HEYS.models.normalizeItemGrams`: `null`, `undefined`, пустая строка и invalid
получают legacy default 100 г; числовая строка или число ≥0 сохраняются, включая
явный `0`. Поэтому неполные nutrients по-прежнему fail-open через defaults, но
нулевая масса больше не превращается в полноценную порцию.

## Основной расчёт инсулиновой волны

Публичная точка входа — `HEYS.InsulinWave.calculate(params)`. Она требует хотя
бы один приём со временем, сортирует такие приёмы и считает текущий статус от
последнего. В расчёт входят состав еды, GL/insulin-index adjustments, форма еды,
контекст активности, время суток, сон, стресс, вода, профиль, NDTE и другие
коэффициенты. Итоговый multiplier ограничен сверху `1.50`.

Для каждого приёма история рассчитывается отдельно. Результат содержит статус,
остаток времени, предполагаемую длительность, phases, историю и overlaps,
персональную базу, IR score и синтетическую трёхкомпонентную Gaussian-кривую.
Это богатый UI view-model, а не сохранённое лабораторное наблюдение.

Внутренний GI волны взвешивается по количеству углеводов. GL складывается по
позициям и дополнительно модифицируется insulin-index rules. Если углеводов нет,
GI получает нейтральный fallback `50`; это техническая устойчивость, а не
утверждение о реальном GI безуглеводного приёма.

## Интеграция с дневником

`computeInsulinWaveData` — React-адаптер DayTab. Он ждёт lazy-модуль через
`heys-insulinwave-ready`, пересчитывается раз в минуту и передаёт в основной
расчёт meals, trainings, профиль и контекст дня. Для прошлого дня расчёт
выполняется на 23:59, затем результат обогащается оценкой ночного окна.

Если основной модуль ещё не загружен, адаптер возвращает fallback только по GI
последнего приёма и базовой длительности. Fallback взвешивает GI по граммам, не
строит историю и использует отдельную таблицу multiplier. Поэтому один и тот же
день может кратковременно показать другое время/статус до события готовности.

Для прошлых дней и ночного окна код оценивает «липолиз» и kcal по времени после
предполагаемого конца волны. При отсутствии приёма следующего дня используется
fallback 08:00; окно ограничивается 16 часами. Эти значения особенно важно
показывать как оценку модели, а не факт с датчика.

## Инварианты

1. Сохранённый meal item должен содержать достаточный snapshot для истории.
2. Сумма приёма проходит через `HEYS.models.mealTotals`, а не дублируется в UI.
3. GI дня и GI волны имеют разные веса и разные назначения.
4. Волна без приёма со временем не рассчитывается.
5. Основной расчёт и DayTab fallback не являются эквивалентными алгоритмами.
6. Персональные и дневные факторы — вход модели, не измерение состояния крови.
7. Верхний cap multiplier должен одинаково применяться к текущей волне и
   истории.
8. Любой user-facing вывод о волне/липолизе должен быть маркирован как оценка.

## Подтверждённые слабые места и пробелы

- Документация заявляет runtime 4.2.5, но source и tests содержат более поздние
  изменения с пометкой v4.3 без смены экспортируемой версии. По версии нельзя
  однозначно восстановить фактическую формулу.
- `DATA_MODEL_NUTRITION.md` смешивает обзор, научные тезисы, формулы и
  changelog. Он полезен для поиска деталей, но слишком широк для навигационного
  канона.
- `INSULIN_WAVE_DOCUMENTATION.md` уже расходится с source: например, описывает
  удалённый `liquidDairyCompensation`, а код и tests подтверждают его удаление.
- DayTab fallback и основной модуль используют разные веса GI и разные наборы
  факторов. Race загрузки способен дать пользователю видимую смену результата.
- Неполные nutrients fail-open через defaults. В результате качество входа не
  видно из итогового статуса и нет общего `confidence/dataQuality` поля.
- Утверждения UI «инсулин высокий», «жир запасается», «липолиз» звучат как
  измеренный физиологический факт, хотя результат полностью расчётный.
- Оценка kcal ночного «липолиза» использует фиксированную скорость от веса и
  fallback следующего приёма 08:00; лабораторной верификации в repository нет.
- Critical test suite имеет ветки, которые тихо завершают test при отсутствии
  internal function. Такой зелёный test не всегда доказывает сам коэффициент.
- Полный набор коэффициентов велик и распределён по нескольким файлам; простой
  счётчик «37/38 факторов» не является стабильным контрактом и уже расходится
  между заголовками docs/tests.

## Как поддерживать без устаревания

- Здесь фиксировать только поток, владельцев, инварианты и подтверждённые риски.
- Формулы и коэффициенты не копировать: ссылаться на source и специализированный
  документ, обновляя его вместе с изменением алгоритма.
- При правке модели проверять одновременно основной расчёт, wave history, DayTab
  fallback, user-facing формулировки и tests.
- Любое изменение смысла должно обновлять честную метку модели и её version, а
  не только комментарий `vX` внутри файла.
