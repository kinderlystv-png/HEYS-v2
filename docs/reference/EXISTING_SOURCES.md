# Ревизия существующих справочников

Проверка выполнена по всем отслеживаемым Markdown-файлам: кандидаты отбирались
по названиям и заголовкам (`reference`, `documentation`, `architecture`,
`context`, `brief`, `summary`, `README`, «справочник», «контекст»,
«архитектура»), затем основные кандидаты читались вручную.

## Вывод

Полезных материалов много, но они решают разные задачи, частично дублируются и
не образуют полный актуальный каталог реализации. Поэтому они сохраняются как
источники для нового справочника, а не объединяются копированием в один большой
файл.

## Обнаруженные противоречия и пробелы

- До ревизии `HEYS_BRIEF.md` называл себя единственным источником истины, хотя
  технические детали распределены между несколькими более узкими документами.
  Формулировка заменена на иерархию проверяемых источников.
- Корневой `README.md` ссылался на отсутствующий `docs/SECURITY.md`; ссылка
  исправлена на действующий `docs/SECURITY_DOCUMENTATION.md`.
- `docs/SUMMARY.md` является ручным индексом, но не перечисляет значительную
  часть действующих справочников и модульных документов.
- Backend-описания часто перечисляют только семь первоначальных API-функций,
  тогда как в каталоге Cloud Functions уже присутствуют дополнительные API, bot,
  cron, backup и maintenance-компоненты.
- Автоматический сайт документации не подключён к действующим материалам и не
  решает задачу актуальности.
