## 3. Где переоценил (замечания ревьюера, по убыванию важности)

### 🔴 R1 — SEC-004 это pre-launch блокер, а не «желательно»

В нашем доке SEC-004 = **P0** (top tenant-isolation, задокументированный
инцидент curator-pollution). Агент оставил warn-only
(`HEYS_WRITE_CONTEXT_STRICT=0` пишет audit, но **пропускает cross-client
writes**) и отнёс в «желательно перед запуском». Для мультитенантного health-app
по нашей threat-model (IDOR/ cross-client = риск №1) это **блокер запуска**. И
это **самый дешёвый** из «нужен го» (флаг после суток soak — агент сам пишет
«SEC-004 trivially»). Вердикт «можно пускать после SEC-020+SEC-006»
под-взвешивает главный риск.

### 🔴 R2 — IDOR не проверен, значит «после запуска» неприемлемо

Сам агент: «❓ IDOR cross-client read блокируется? — нужны 2 credential пары».
Запуск мультитенантного health-app без **проверенной** (не предположенной)
изоляции чтения между клиентами = незакрытый главный вопрос. L3 IDOR-верификация
— **до** публичного запуска, не «в первый месяц». Нужны 2 PIN/phone
(эскалировано верно, но как pre-launch).

### 🟡 R3 — L6 (152-ФЗ / DR) для РФ перс-данных — не «может подождать»

Restore-drill, retention, deletion-cascade, RPO/RTO — юридическая база для
перс-данных; L6 «не начат». Минимум baseline до запуска.

### 🟡 R4 — «65-70% готовности» — необоснованная метрика

Методология % не вводила. Finding-count % вводит в заблуждение: 1 открытый P0 ≠
1 открытый P3, знаменатель 20 = «найдено пока», не «вся поверхность». Заменить
на **список открытых P0/P1 + список непройденных слоёв**.

### 🟡 R5 — «production-ready: да» — не зона агента

Агент отдаёт **risk-ledger**; go/no-go — решение пользователя на его основе.
Само-сертификация «можно пускать» выходит за роль.

### Мелкое

- SEC-010 «может подождать» — оптимистично без триажа. 5 critical в runtime →
  сначала триаж (какие CVE, достижимы ли), потом решать.
- SEC-008 «accepted-risk» — ок, **если** источник FAQ-контента реально прослежен
  как статический (не предположен). Подтвердить, что трассировка была.

## 4. Рекомендованный порядок (reviewer)

1. **SEC-020 активация** (5 мин: env-var + setWebhook) — до этого fake-updates
   принимаются.
2. **SEC-006** — единственное 🔴-активно-эксплуатируемое. Быстрый фикс:
   `yc storage bucket update heys-photos --anonymous-access-flags read=false`;
   signed URLs потом. До появления реальных фото.
3. **SEC-004** — pre-launch: сутки Phase B → включить strict. Самый дешёвый P0.
4. **L3 IDOR-сессия с 2 cred-парами** — до запуска. Главный незакрытый unknown.
5. **SEC-010** — триаж 5 critical (CVE, reachability) → потом решать «подождёт».
6. **L6 152-ФЗ/DR baseline** — restore-drill + retention + deletion-cascade.
7. Post-launch: SEC-017 RLS (defense-in-depth), SEC-011 MFA, L8 пентест.

---

## 5. Как процессно лучше

- Убрать единый «% готов» → «открытые P0/P1 + непройденные слои».
- Каждое «live-verified» → воспроизводимая команда в Facts Table.
- Агент не само-сертифицирует «production-ready» — отдаёт risk-ledger.
- Взвешивать severity × exposure, не считать находки.
