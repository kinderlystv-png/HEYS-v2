## Контекст продукта

HEYS — серьёзный продукт нутрициологии и сопровождения, разрабатывался ~9
месяцев и тысячи часов. Целевая аудитория — взрослые люди, которые устали от
диет и хотят системного подхода. Тариф Pro — 7 990 ₽/мес. Это
**премиум-продукт**, не утренняя зарядка-приложение.

Из этого следуют требования к копирайту: **серьёзный, спокойный,
доказательный**. Не маркетинговый-крикливый, не разговорный-приятельский, не
science-flex для своих.

## 🚫 Чёрный список — никогда в копирайте

### Разговорный сленг и стенограмма реплик пользователя

Реплика пользователя в чате со мной — это **стенограмма** для меня, не финальный
копирайт. Перевожу смысл в нормативный регистр всегда.

| Не использовать                      | Заменить на                                         |
| ------------------------------------ | --------------------------------------------------- |
| «плотно»                             | «детально», «углублённо», «максимально внимательно» |
| «круто», «классно», «норм», «зайдёт» | конкретное описание выгоды                          |
| «закрывать» (= handle)               | «брать на себя», «выполнять»                        |
| «кидать», «скидывать» (= send)       | «отправлять», «присылать»                           |
| «фишка», «штука»                     | «функция», «возможность»                            |
| «фигачить»                           | в копирайте — никогда                               |

### Стартап-жаргон и хайповые слова

Допустимо только в внутренних обсуждениях, не в клиентском копирайте.

| Не использовать      | Заменить на                             |
| -------------------- | --------------------------------------- |
| «суперапп»           | «полноценное приложение», «всё в одном» |
| «киллер-фича»        | «ключевая возможность»                  |
| «вау-эффект»         | конкретный результат                    |
| «продукт-маркет фит» | вообще не упоминать                     |

### Кальки с английского

| Калька                                        | По-русски                             |
| --------------------------------------------- | ------------------------------------- |
| «закрывают наши алгоритмы» (close = handle)   | «берут на себя», «выполняют»          |
| «пример как звучит» (example of how X sounds) | «пример того, как звучит» (с запятой) |
| «out of the box»                              | «сразу из коробки», «с самого начала» |

### Технические детали, которые клиенту неважны

На лендинге — взгляд клиента, не взгляд разработчика/процесса.

| Что писать не надо                                                          | Почему                                                                                                    |
| --------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| «БЖУ», «микроэлементы», «термогенез» в основных карточках                   | Научные термины оттолкнут массовую аудиторию. Допустимы в углублённой Navigator-секции, где раскрываются. |
| «куратор берёт заявку, создаёт клиента»                                     | Внутренний процесс. Клиенту неважно.                                                                      |
| «настраивает приложение: цели по калориям и БЖУ, водный баланс, тренировки» | Мелкая декомпозиция = шум. Достаточно «настраивает приложение под вашу цель».                             |
| «PIN», «scoped key», «overlay»                                              | Технические детали реализации.                                                                            |

**Правило:** клиенту важно — **что произойдёт с ним** и **что от него
ожидается**. Не — как это устроено внутри.

### Overpromise — обещания, которые не подкрепляются продуктом

| Не обещать                         | Реальность                                                      |
| ---------------------------------- | --------------------------------------------------------------- |
| «куратор отвечает в течение часа»  | Нет SLA. Куратор — живой человек, скорость зависит от загрузки. |
| «куратор берёт на себя 98% работы» | Преувеличение. Клиент всё равно присылает данные и участвует.   |
| «куратор ведёт дневник за вас»     | Допустимо только для Pro: куратор переносит присланные данные.  |

### Структурные шумы

- **Дублирование trust-claims.** Если «без карты», «без автосписаний», «не
  медицинская услуга» уже есть в Trial/FAQ — не повторять в
  Hero/CTA-микротексте.
- **Мелкие микротексты под CTA**, если они отвлекают и дублируют то что уже было
  ниже.
- **Устаревшая нумерация секций.** Бейджи «01 — Ваша ситуация» когда секция уже
  не первая — выглядит как brain-fart.

### Тактичность клинических терминов

В контексте веса, здоровья и тела клиент уязвим. Слова, которые звучат
objectifying или judgemental, ломают доверие — даже если технически верные.
Выбирай **холистичные, клинические термины**.

| Лучше избегать                     | Тактичнее                                                            |
| ---------------------------------- | -------------------------------------------------------------------- |
| «тело» (особенно в контексте веса) | «организм» — биологическая система, не объект для ремонта            |
| «лишний вес»                       | «вес» (без оценочной приставки)                                      |
| «жирный», «полный»                 | в копирайте — никогда                                                |
| «недоедание / переедание»          | «дефицит / избыток калорий»                                          |
| «срыв» (в смысле «провал клиента») | «срыв» как нейтральный термин — OK; но без обвинительной коннотации  |
| «безволие», «лень»                 | в копирайте — никогда (см. NavigatorSection: «сила воли ни при чём») |

**Принцип:** В нише weight-loss / wellness копирайт постоянно касается
чувствительных тем. Базовое правило — **не давать клиенту повод почувствовать
себя проблемой**. Описывай биологию и систему, а не «провалы» клиента.

### Защитные конструкции

В премиум-продукте базовые сервисные действия **не подчёркиваются**. Они
подразумеваются.

| Защитно (звучит дёшево)                                | Уверенно                                                                                   |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------ |
| «куратор сам пишет вам»                                | «куратор знакомится с вами»                                                                |
| «приложение само распознаёт»                           | «приложение распознаёт»                                                                    |
| «реальный живой человек, не бот» (без контекста)       | в контрастной формуле — да; одиночно — лишнее                                              |
| «настоящие данные, не выдумка»                         | «ваши данные за неделю»                                                                    |
| «смотрит результаты анализов, если они у вас уже есть» | «анализы и обследования» (как часть списка клиентских данных)                              |
| «существующие/имеющиеся анализы»                       | «анализы» — прилагательное само по себе хедж + логически абсурдно («есть несуществующие?») |

**Под-паттерн: оправдательное «если у вас уже есть» / «если они доступны» /
прилагательные «существующие», «имеющиеся», «доступные»** — всё это один класс
защитных конструкций. Каждое слово, которое подсвечивает «может быть, а может и
нет», имплицирует негативный сценарий («если нет — неловко получилось»).
Особенно нелепо звучат прилагательные типа «существующие» — они логически
избыточны (в природе нет «несуществующих» анализов).

**Правило:** Если опциональные данные перечислены в списке клиентских данных,
**контекст уже подразумевает «то что есть у клиента»**. Никакого хеджа не нужно.
Просто: «анализы и обследования» — как часть списка.

Если в каком-то контексте действительно нужно подчеркнуть, что данные не
обязательны — это **другая работа** в копирайте (например, в FAQ или
дисклеймере), а не в основном описании сервиса.

**Принцип:** если действие — это **сервисная норма** продукта, описывай его как
факт без оправдания. «Сам/само» подсвечивает то, что должно быть очевидным —
имплицирует, что бывает иначе.

## ✅ Делать

### Клиентоориентированный flow

В сервисной модели **инициирует тот, кто предоставляет сервис**. Клиент сделал
свой шаг (оставил заявку, оплатил) — дальше отрабатывает HEYS.

| Не клиентоориентированно         | Клиентоориентированно                                 |
| -------------------------------- | ----------------------------------------------------- |
| «Пишете куратору»                | «Куратор сам пишет вам»                               |
| «Заходите в раздел Дневник и...» | «В дневнике вы увидите...»                            |
| «Загрузите фото еды»             | «Сфотографируйте еду — приложение распознает её само» |

### Конкретика вместо абстракции

| Слабо                | Сильно                                                        |
| -------------------- | ------------------------------------------------------------- |
| «Шаг 1: Вижу»        | «День 1 — знакомство с куратором»                             |
| «Простой процесс»    | «От первого сообщения до устойчивого режима — за месяц»       |
| «Глубокая аналитика» | «Навигатор показывает что мешает в вашей конкретной ситуации» |

### Грамматика и регистр

- **Везде Вы-форма.** Никаких миксов с «Ты». Если хоть одна секция в «Ты» —
  поправь.
- **Запятые перед** «как», «который», «что» в придаточных. «Объясняет, как
  пользоваться» (а не «объясняет как пользоваться»).
- **Тире вместо двоеточия** там, где идёт пояснение: «Куратор остаётся доступен
  — всё время подписки».
- **«Не бот, не FAQ»** — короткое отрицание-формула работает в подзаголовках для
  контраста.

### Замена научных терминов на «бытовой» язык

В местах, где научная точность не нужна для убеждения, но термин может
оттолкнуть — переводи на бытовой:

| Научно                                  | Бытово (для широкой аудитории)               |
| --------------------------------------- | -------------------------------------------- |
| «снижает термогенез»                    | «тормозит похудение даже в дефиците»         |
| «недосып держит воду»                   | «из-за недосыпа вес стоит»                   |
| «дефицит белка ведёт к вечерним срывам» | «вечерние срывы — часто нехватка белка днём» |

В Navigator-секции, где **раскрывается** уникальность системы, научные термины
уместны — там цель показать глубину.

### 2026-05-18 — Карточка «День 1» без профессионального substance

**Я написал** (после трёх итераций):

> «Куратор связывается с вами в первый день — в мессенджере, голосовым или
> видеозвонком, как вам удобнее. Разбирается в вашей цели, прошлом опыте, в том
> что мешает сейчас. В результате у вас с куратором — полная картина и план на
> первую неделю.»

**Реакция:**

> «куратор интересуется всякими аспектами, хоть мы не медицинской услуга, но для
> эффективной оценки состояния организма и работы как эксперт должен применять
> различные методологии — обращаться к анализам, исследованиям, базовые тесты по
> биомеханике делать. Какое-то взаимодействие более плотное должно быть. Там не
> просто разговор, там заводится какая-то карточка. Надо как-то это все описать
> по-серьезному.»

**Проблема:** Карточка описывала контакт как «поговорили — разошлись». Это
снижает воспринимаемый профессиональный уровень. У эксперта/куратора первый
контакт — это **profession intake**: оценка состояния, сбор данных, заведение
карточки клиента. Без явных артефактов профессиональной работы сервис read'ится
как «дружеская консультация», а не «методика с протоколом».

**Принцип:** Когда описываешь профессиональный контакт со специалистом,
**показывай артефакты экспертизы**:

- Конкретные данные, которые специалист собирает (не «обсудили», а «собрал в
  карточку: цель, прошлый опыт, особенности тела, образ жизни»)
- Использование внешних данных (анализы, обследования) — даже с дисклеймером
  «если у вас уже есть»
- Явное упоминание **карточки клиента** или другого артефакта-записи — поднимает
  воспринимаемую серьёзность
- Балансировать с дисклеймером «не медицинская услуга»: куратор **использует**
  анализы клиента, **не назначает** их

**Стало:** «Куратор связывается с вами в первый день — в мессенджере, голосовым
или видеозвонком, как вам удобнее. Собирает данные для вашей мета-карты: цель,
прошлый опыт, образ жизни, особенности организма, существующие анализы и
обследования. Смотрит результаты анализов и обследований, если они у вас уже
есть. В результате у вас с куратором — полная картина и план на первую неделю.»

### 2026-05-18 — «куратор пишет в мессенджере» — загнали клиента в одну модальность

**Я написал** (в карточке «День 1» после двух итераций):

> «Куратор пишет вам в мессенджере. Разбирается в вашей цели...»

**Реакция:**

> «логично, наверное, как-то обозначить, что здесь есть вариации, потому что
> кому-то может быть нужно больше — у клиента может быть потребность в том,
> чтобы более тесно взаимодействовать, тогда куратор может созвониться, по
> видеосвязи или по телефону. А кому-то удобнее минимизировать общение. ...
> человек может подумать "как можно переписки так много всего обсудить, это же
> долго"»

**Проблема:** Указание конкретной модальности взаимодействия («пишет в
мессенджере») — это **artificial constraint**. Реально куратор гибок: текст,
голос, видео. Жёстко зафиксированная модальность вызывает у читателя:

1. Снижение воспринимаемого уровня сервиса (текст feels lower-touch чем созвон)
2. Friction для клиентов с другими предпочтениями («а я не люблю писать»)
3. Сомнения в эффективности («как столько всего обсудить перепиской?»)

**Принцип:** Если у продукта есть гибкость в формате/канале/времени —
**показывай эту гибкость в копирайте**. Не своди к одной конкретной модальности,
когда их несколько.

Формула: «X — в формате A, B или C, как вам удобнее» оставляет агентство
клиенту, не прячет его за «решит сервис».

**Стало:** «Куратор связывается с вами в первый день — в мессенджере, голосовым
или видеозвонком, как вам удобнее. Разбирается в вашей цели, прошлом опыте, в
том что мешает сейчас. В результате у вас с куратором — полная картина и план на
первую неделю.»

**Lexical sub-lesson (в процессе правки):** Первая попытка списка каналов была
«голосом или по видео» — кривая параллель: «голосом» (творительный падеж,
awkward как название канала) + «по видео» (предлог по + дательный). Плюс
«голосом» в значении канала режет ухо. **Правильно:** все элементы списка в
одном грамматическом ключе. «Голосовым звонком» / «видеозвонком» / «по
видеосвязи» — нормативные русские термины. В списках каналов держи параллельную
структуру (все instrumental, либо все «по + X»).

**⚠ Adjacent issue:** В `CuratorSection.tsx` карточка «Обычная переписка»
имплицитно тоже сводит сервис к тексту. Если у куратора фактически есть
видео/голос — там тоже надо отразить (или переименовать карточку в «Общение в
вашем темпе и формате»).

### 2026-05-18 — «к концу разговора» при «куратор пишет»: внутреннее противоречие + лишний temporal marker

**Я написал** (в карточке «День 1» после первой итерации):

> «Куратор пишет вам в мессенджере... К концу разговора у вас есть конкретный
> первый шаг и план на первую неделю.»

**Реакция:**

> «фраза к концу разговора — странная. во первых куратор ПИШЕТ а значит нет
> разговора. во вторых это какая то не нужная деталь. может лучше написать что
> то типа "в результате у вас с куратором будет полная картина и так далее"»

**Две проблемы:**

1. **Лексическая несогласованность.** «Куратор **пишет** в мессенджере»
   (переписка) + «к концу **разговора**» (устное) — внутреннее противоречие.
   Слова из разных регистров в одной мысли. Всегда проверять: совпадает ли
   модальность действия (письменно/устно/видео) во всех частях описания.

2. **Временной маркер вместо состояния.** «К концу разговора у вас есть X» — это
   **точка во времени**, процессное мышление. Сильнее работает **состояние как
   результат**: «у вас с куратором — полная картина». Без временного маркера
   фраза становится тверже: это **гарантированный исход**, а не «когда-то к
   концу».

**Принцип:** Результат подавай **как состояние**, не как точку во времени. «У
вас есть Х» сильнее чем «к концу Y у вас есть Х». Формула «у вас с куратором»
(партнёрство) сильнее чем «у вас» (односторонне) — особенно в карточках про
сервисное взаимодействие, где партнёрство — ключевая ценность.

**Стало:** «Куратор пишет вам в мессенджере. Разбирается в вашей цели, прошлом
опыте, в том что мешает сейчас. В результате у вас с куратором — полная картина
и план на первую неделю.»

### 2026-05-18 — Карточка «День 1» без концовки-результата

**Я написал** (в `HowItWorksSection.tsx`, карточка «День 1»):

- Headline: «Знакомство с куратором»
- Body: «В первый же день куратор знакомится с вами в выбранном мессенджере.
  Помогает разобраться в системе, настраивает её под вашу цель. От вас — только
  рассказать о себе.»

**Реакция:**

> «день первый — очень плохо написана карточка объективно»

**Что было не так (4 объективных проблемы):**

1. **Headline дублирует label.** Label «ДЕНЬ 1» + headline «Знакомство с
   куратором» — одно и то же. Headline не несёт нового смысла, занимает место.
2. **Body повторяет headline.** «В первый же день куратор знакомится» —
   тавтология после «Знакомство с куратором».
3. **«Помогает разобраться в системе» — пустота.** Без конкретики что именно
   даст это «разобраться». Клиент не понимает что он получит.
4. **«От вас — только рассказать о себе» — защитная концовка.** Мол, не
   пугайтесь, легко. Это оборона, не подача. Карточка завершается на низкой
   энергии.

**Принципы:**

- **Headline ≠ label.** Если label = «ДЕНЬ 1», headline должен дать **другой**
  смысл (контраст, выгоду, обещание). Например, «Личный разговор, не анкета» —
  отражает ожидание клиента «опять форму заполнять» и сразу его опровергает.
- **Body не должно открываться повтором headline.** Первая фраза =
  действие/факт, а не пересказ заголовка.
- **Карточка должна заканчиваться РЕЗУЛЬТАТОМ.** Что клиент получит к концу
  этого этапа? Конкретный deliverable («первый шаг», «план на неделю»,
  «понимание ваших цифр»). Не «вам станет легче», а «у вас будет X».
- **Защитные концовки в духе «не бойтесь, всё просто» — слабая энергия.**
  Премиум-продукт **подаёт** сервис, а не оправдывает его лёгкость.

**Стало:**

- Headline: «Личный разговор, не анкета»
- Body: «Куратор пишет вам в мессенджере. Разбирается: какая цель, что пробовали
  раньше, что мешает сейчас. К концу разговора у вас есть конкретный первый шаг
  и план на первую неделю.»

### 2026-05-18 — «куратор САМ пишет» — защитная конструкция

**Я написал** (в `HowItWorksSection.tsx`, карточка «День 1»):

> «Сразу после регистрации куратор сам пишет вам в выбранном мессенджере»

**Реакция:**

> «это же странноватый оттенок. звучит дешево. зачем акцентировать внимание на
> том что сам это делает?»

**Принцип:** В премиум-продукте базовая сервисная норма **не подчёркивается**.
«Сам» работает как защитная конструкция — как будто продукт оправдывается
«да-да, реально человек пишет». Это похоже на ресторан, который пишет «официант
сам принесёт вам блюдо» — звучит дёшево, имплицирует низкое baseline-ожидание.

**Правильно:** «В первый же день куратор знакомится с вами в...» — действие
подаётся как естественное, без оправдания.

**Расширение принципа:** То же относится к «приложение само распознаёт»,
«система сама считает», «куратор сам напомнит» — везде, где «сам/само»
подсвечивает то, что **по дизайну продукта** обязательно. Если контраст в духе
«без вашего участия» уже задан контекстом — «само/сам» лишнее.

## 🛠 Как пользоваться этим документом (для агента)

**Перед написанием/правкой любого user-facing текста в `apps/landing/`:**

1. **Прочитать чёрный список** — проверить себя на anti-patterns (сленг, жаргон,
   кальки, технические детали, overpromise, шумы).
2. **Прочитать «Делать»** — применить принципы (клиентоориентированный flow,

3. **Бегло пробежать «История замечаний»** — освежить контекст конкретных
   провалов прошлого.
4. **Применить «правило стенограммы»:** если фраза скопирована из реплики
   пользователя дословно — это сигнал, что я не переработал смысл. Перепиши
   собственными нормативными словами.

   - **Принцип** (правило на будущее)
2. Если замечание выявило **новый паттерн** — обновить «Чёрный список» или
   «Делать».
