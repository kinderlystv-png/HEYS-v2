# Карта реализации (методология → код)

Трассировка «каждая единица методологии → как ложится в код». Цель — чтобы при
реализации **ничего не потерялось**: пустая ячейка в матрице видна сразу.

**Это не пересказ методологии**, а ключ-таблица по стабильным ID. Содержание
«что/зачем» живёт в [`METHODOLOGY.md`](METHODOLOGY.md); схема данных, контракты
ядра и формулы — в [`CONSTRUCTOR_SPEC.md`](CONSTRUCTOR_SPEC.md); здесь — связка
«единица → реализация → модуль → очередь» плюс трёхосевой статус готовности.

**Состояние проекта: source-реализация готова.** Код доменного слоя режима
мобильности создан в `apps/web/mobility/`: каталоги, safety-валидаторы, builder,
runner, assessment, onboarding, readiness, records/progression, bibliography,
React UI, public entry и lazy stub. Generated bundle/legacy-sync/browser QA не
запускались без отдельной команды. «Заполн.» в трейс-таблицах ниже — про
авторство _этой карты_, а канон фактической реализации — в
`apps/web/mobility/IMPLEMENTATION_PROTOCOL.md`.

**Архитектура: ядро + контент (правило проекта, CLAUDE.md/AGENTS.md «Архитектура
тренировочных режимов»).** Этот модуль — **вторая реф-реализация** после
пальцев, строится **против контрактов общего ядра**, а не как самостоятельная
копия. Предлагаемые ниже модули — это **доменный слой** (контент-данные + тонкий
код), который мапится на канонический словарь ядра: `axis_catalog`≈kernel
`qualities`, `atom_catalog`≈kernel `catalog`,
`routine_builder`/`mode_engine`≈kernel `session-builder`/`periodization`,
`population_gating`≈kernel `gating`, `validators`≈kernel
`validator-framework`+доменные правила. Тривиально-общее (роутер, timer-shell,
records, bibliography, readiness-математика, валидатор-фреймворк, онбординг)
переиспользуется сразу; тяжёлые общие движки (периодизация, лимитер, прогрессия)
**выносятся в `apps/web/_kernel/` после этого домена** (правило двух/трёх).
Сугубо доменное (вис-таймер vs дыхание/гониометрия) остаётся в
`apps/web/mobility/`.

**Релиз без MVP (правило проекта, CLAUDE.md / AGENTS.md).** Модуль строится
**сразу в полной релизной версии**, как режим пальцев: **ничего из методологии
не урезается и не откладывается** как условие релиза. «Очередь» ниже — только
**порядок сборки** (зависимости), а не тиры урезания фич: всё перечисленное
входит в релиз.

**Оси статуса (как в режиме пальцев):** **Метод.** (наука/логика методологии) ·
**Движок** (логика конструктора) · **UI** (пользовательский поток). Легенда: ✅
готово · 🟡 частично · ⬜ не начато · — n/a.

**Колонки трейс-таблиц:** ID · что (методология, кратко) · реализация (что
станет в коде) · модуль(и) · **очередь** (порядок сборки внутри одного релиза: 0
метод. / 1 раньше / 2 позже — **не урезание**, всё в релиз) · **заполн.** (✅
строка заполнена · 🟫 черновик · ⬜ пусто).

**Синхронизация.** Канон трёхосевого статуса — в этом файле (📍 Сводка
готовности); ручной зеркальный mirror — `METHODOLOGY.md §11.1`. Coverage-guard
подключён: `pnpm --dir apps/web check:mobility-map` проверяет строки карты,
Q-пул и обязательные source/test артефакты. Правишь методологию → запускай guard
и проверяй, что у новой единицы есть строка здесь.

> **Статус карты:** заполнена целиком — на каждую единицу методологии (1.1–1.8,
> part-2, 3.1–3.6, block-A…J, 5.0–5.10, режимы 6, 7.1–7.4, 8.1–8.4, 9.1–9.8,
> S1–S9, part-0/10/11) есть строка. Дальнейшая работа — **не «заполнение карты»,
> а решение открытых вопросов** (пул ниже) и собственно реализация полного
> релиза в порядке очереди сборки.

- [x] `mode_engine` — **все 7 режимов** (утро / pre-workout RAMP / post-workout
      / развитие мобильности / вечер-релакс / реабилитация-рамка /

      ручной ввод углов + joint-by-joint лимитер (`Q-test-1/2` ✅, `Q-limit-1`
      🟡).
- [x] `readiness` — HRV/субъективные шкалы как мягкий вход (§8.3, опц. сигнал).
- [x] Онбординг (**4 входа**: уровень + популяция-флаги + цель + оборудование) +

      поверх `recordsStore`/`progression.romTrend`.

**Порядок сборки (зависимости, всё в одном релизе):** safety (S1–S9) →
`axis_catalog`/`atom_catalog`+`equipment_profile` →
`routine_builder`/`mode_engine` → `routine_runner`/`breath_runner` →
`mobility_assessment`/онбординг → `readiness` → UI/entry/lazy integration.

**Возможные улучшения сверх методологии (не урезание релиза, а будущая
полировка):** камера-гониометрия вместо ручного ввода углов; перцентильная
аналитика ROM-прогресса; видео-контент атомов. Это _добавки поверх_ полного
релиза, а не вынесенные из него разделы.

**Числовой guardrail (держать при реализации):**

- **Safety-baseline:** S1 (cold-max блок), S2 (pain-стоп), S3 (warmup-гейт), S9
  (утренний end-range) — fail-closed, 0 регрессий в safety-тестах.
- **Доза:** атом без `doseConfidence` не грузится; «нет оптимума» (5.3) → фикс.
  дефолт + правило прогрессии «пока даёт прирост», не автонакрутка объёма.
- **Автономный вектор:** в одной сессии не смешиваются `autonomic:тонизация` и
  `autonomic:релакс` (инвариант 1.2/1.7).

# Детальные карточки реализации

Развёрнутая «схема реализации» по ключевым единицам. Однородные группы (блоки,
уровни, S-правила) — групповыми карточками. Открытые вопросы в карточке — ссылки
на `Q-*` из пула.

## Карточка 1.2 — Цель × время (организующий хребет)

**Методология §1.2:** один жест = четыре протокола в зависимости от цели
(подготовка/развитие/восстановление/регуляция); путать = делать не то.

### M1 — ось `purpose` (данные, очередь 1)

- Поле атома `purpose: 'prep' | 'develop' | 'recover' | 'regulate'` — первичная
  ось пула (аналог `energySystem` у пальцев). Задаёт дозу, темп, отдых,
  совместимость.
- Поле `autonomic: 'tonify' | 'neutral' | 'relax'` — автономный вектор (1.7).

### M2 — инвариант сессии (валидатор, очередь 1)

- `V_autonomicCoherence(session)`: в одной сессии не смешивать `tonify` и
  `relax` → warn/reorder. Реализует «не путать сигнал нервной системе».
- `V_developSeparation`: длинная развивающая статика (`purpose:develop`,
  hold>60с) не в разминочной сессии (`purpose:prep`) → переставить (связь S8).

### M3 — выбор по режиму (очередь 1, в `mode_engine`)

- Режим (часть 6) фиксирует целевой `purpose`+`autonomic`, `routine_builder`
  подбирает атомы только совместимого вектора.

**Тесты:** сессия `[tonify+relax]` → warn; develop-статика в prep-сессии →
reorder; режим «вечер-релакс» не пускает `autonomic:tonify`.

**Модули:** `atom_catalog` (поля), `validators`, `routine_builder`,
`mode_engine`. **Очередь:** 1. **Вопросы:** `Q-mode-1` (автоген vs ручной
набор).

## Карточка 1.4 — Лимитер подвижности (joint-by-joint)

**Методология §1.4 + §8.2:** найти, где потерян мобильный сустав; мобилизировать
тугое, стабилизировать разболтанное; различать потолок/контроль/силу.

### M1 — скрин → отклонения (очередь 1)

- `mobility_assessment` собирает скрины (8.1), сравнивает с нормой по суставу/
  возрасту (нужны нормы — `Q-norm-1`).

### M2 — классификатор типа (очередь 1–2, БЛОКЕР)

- `limiterAudit(screens)` для каждого тугого сустава определяет тип: **потолок**
  (пассив↓ → растяжка/PNF/нагруж.), **контроль** (разрыв пассив−актив →
  end-range сила/CARs), **сила/техника** (не сюда). Формула — `Q-limit-1`.

### M3 — веса приоритетов блоков (реализовано)

- Ведущий лимитер → подъём соответствующих блоков в `routine_builder`; один
  ведущий на период; переоценка по ретесту (8.4).

**Тесты:** тугой голеностоп + малый разрыв → приоритет растяжка голени; большой
разрыв пассив−актив → end-range сила, не растяжка.

**Модули:** `mobility_assessment`, `routine_builder`. **Очередь:** 1–2.
**Вопросы:** `Q-limit-1` (формула, блокер), `Q-norm-1` (нормы ROM).

## Карточка 3.1 — Тепло-сосудистый эффект + циркадность

**Методология §3.1:** разогрев = ↑темп/кровоток/амплитуда; гибкость минимальна
утром, пик к вечеру → утром полнее разминка, end-range позже.

### M1 — warmup-эффект (очередь 1)

- Ramp-up как обязательная процедура перед интенсивным/конечно-диапазонным (S3).

### M2 — циркадная коррекция (очередь 1 мягко / 2 полно)

- `circadianAdjust(timeOfDay)`: утром → удлинить разминку, опустить целевой
  end-range; `S9` warn на глубокий end-range утром без полной разминки.
- В релиз: мягкий reason + S9; утренний режим и недельный planner учитывают
  контекст (`Q-mode-2`).

**Модули:** `mode_engine`, `validators`, `routine_runner`. **Очередь:** 1–2.
**Вопросы:** `Q-mode-2` (учитывать ли время суток — да, в релиз).

## Карточка 6 — режимы и периодизация (mode_engine)

**Методология ч.6:** 7 режимов по «момент × автономный вектор» + периодизация
мобильности + тайминг холода + сквозные инварианты.

### M1 — режимы (очередь 1–2)

- `mode_engine.buildSession(mode, profile, equipment)` → набор блоков под
  целевой `purpose`+`autonomic` режима. **Все 7 режимов в релиз:** утро /
  pre-workout (RAMP) / post-workout / развитие мобильности / вечер-релакс /
  реабилитация-рамка / анти-седентарность.

### M2 — периодизация (реализовано)

- Развитие ROM в базовые фазы, поддержание в пик; F не впритык к ключевой
  нагрузке; deload — лёгкая мобильность.

### M3 — тайминг холода (реализовано)

- CWI/холод — только «свежесть завтра», не после адаптивной силовой (3.4);
  advisory, не гейт (`Q-cwi-1`).

### M4 — сквозные инварианты (очередь 1)

- Один автономный вектор/сессию (1.2/V_autonomicCoherence); развитие отдельно от
  разминки; «микро > эпизод» (подсказка регулярности).

**Модули:** `mode_engine`, `routine_builder`, `validators`. **Очередь:** 1
(базовые режимы+инварианты), 2 (периодизация/холод/реабилитация) — всё в релиз.
**Вопросы:** `Q-mode-1`, `Q-mode-2`, `Q-cwi-1`.

**Модули:** `population_gating`, `validators`, `mode_engine` (акценты).

# Заметки целостности — дома «бездомным» механизмам

Механизмы методологии, которым легко не найти дом при реализации — назначаем дом
заранее, чтобы не уронить:

- **§2.5 карта эффектов (8 областей)** → `effectMap` справочник + honest-UI
  экран; не только текст методологии, а гейт ожиданий (1.6). Дом:
  `atom_catalog.effectMap`.
- **§3.1 циркадность** → не «факт в прозе», а `circadianAdjust` + S9. Дом:
  `mode_engine`/`validators`. Без дома легко потерять (это новое vs пальцы).
- **§3.4 тайминг холода** → `mode_engine` advisory-правило, не гейт; легко
  забыть, т.к. «холод» вне атомов. Дом помечен (`Q-cwi-1`) и показан в UI.
- **§1.5 разрыв пассив−актив** → не только метрика теста, а вход в подмешивание
  end-range силы. Дом: `mobility_assessment`→`routine_builder`.
- **§5.3 «нет чёткого оптимума дозы»** → не баг, а решение подачи: фикс.
  дефолт + прогрессия «пока даёт прирост». Дом: `atom_catalog` дефолты +
  `records` прогресс. Не превращать в автонакрутку объёма (риск 1.8).
  `Q-dose-1`.
- **§7.4 контекст «офис/сидячий»** → не отдельный продукт, а акцент-профиль
  (анти-седент. + грудной/бёдра). Дом: `population_gating` акценты.
- **Терминология:** `purpose`/`autonomic`/`axis` — стабильные имена тегов атома;
  зафиксировать в схеме §5.0 (`Q-atom-1`), не плодить синонимы.
