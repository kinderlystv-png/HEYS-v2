## Часть 1. Конструктор тренировок

Формальная модель. Цель — чтобы из этих сущностей собиралась **любая** валидная
тренировка под дисциплину/уровень/цель, а невалидная — блокировалась. Все
ограничения методологии ч.9 (безопасность) входят сюда как обязательные
валидаторы (§1.6).

### 1.1. Иерархия

```
Атом (Exercise/Drill)
  → Блок (Block)            // несколько атомов одного качества + параметры
    → Сессия (Session)      // блоки одного дня + разминка + заминка
      → Микроцикл (Week)    // сессии недели, баланс нагрузка/восстановление
        → Мезоцикл (Meso)   // 3–6 недель, фокус, deload
          → Макроцикл (Macro) // сезон к цели, выбор модели периодизации
```

### 1.2. Атом (Exercise)

Минимальная единица. Расширяет существующий формат `exercises[i]` HEYS, обобщая
его с «только фингерборд» на любую модальность:

```
Exercise {
  id
  modality        // 'fingerboard'|'board'|'wall'|'campus'|'weights'|'drill'|'mobility'|'antagonist'
  quality         // ось из методологии ч.2: 'finger_strength'|'max_strength'|'power'|
                  //   'anaerobic_capacity'|'aerobic_base'|'technique'|'antagonist'|'mobility'|'mental'
  energySystem    // 'phosphagen'|'glycolytic'|'aerobic'|null

  // — ФОРМА ДОЗЫ (находка пула #2): доза бывает разной природы —
  doseShape       // 'hang'|'attempts'|'circuit'|'continuous'|'reps'|'process'
  dose            // объект, поля зависят от doseShape (см. ниже)

  // — НАГРУЗКА (находка пула #1): расширенный loadModel —
  loadModel       // 'addedWeightKg'|'pctMax'|'rpe'|'grade'|'bodyweight'|'rm_margin'|'none'
  loadValue       // число в смысле loadModel; для rm_margin — секунды запаса до отказа
                  // для grade — enum (находка #3.2): 'easy-mid'|'submax'|'near-limit'|
                  //   'limit'|'limit-skill'; для none — null (тактика)

  // — опционально, по модальности (находка пула #3) —
  gripId?         // только fingerboard/board: ссылка в GRIPS
  edgeSizeMm?     // только fingerboard

  // — мета —
  minLevel        // 'beginner'|'intermediate'|'advanced'|'elite'
  minAge          // число | null. null = НЕТ ДАННЫХ → fail-closed (часть 9).
                  // явное «без гейта» кодируется 0 (находка пула #4)
  dangerLevel     // 'low'|'moderate'|'high'|'very-high' (как у GRIPS)
  sourceIds[]     // привязка к bibliography (СИЛА ИСТОЧНИКА живёт там)
  doseConfidence  // 'A'|'B'|'C' — доказанность ИМЕННО ЭТИХ ЧИСЕЛ (сек/подходы/
                  // отдых/вес). Отдельно от силы источника: протокол может
                  // опираться на 🟢 A работу, но иметь экстраполированную дозу → 'B'.
}
```

**Две оси доказательности (не путать):**

- _Сила источника_ — в [`BIBLIOGRAPHY.md`](BIBLIOGRAPHY.md), свойство ссылки.
- _`doseConfidence`_ — свойство протокола: насколько доказана его дозировка.
  Пример: `pe_fingerboard_lactic` → sourceIds:[`maciejczyk2022` 🟢] но
  `doseConfidence: B` (числа интервалов — экстраполяция из физиологии).

**`doseShape` → поля `dose`** (находка #2 — доза не всегда «вис × подход»):

```
hang:       { workSec, restSec, reps, sets, restSetsSec }      // фингерборд
attempts:   { movesPerAttempt:[min,max], attempts:[min,max], restSetsSec } // лимит-болдер, кампус
circuit:    { problemsPerRound, rounds, restRoundsSec }        // 4×4, link-ups
continuous: { workSec, sets }                                   // ARC
reps:       { reps:[min,max], sets, restSetsSec }               // антагонисты, мобильность
process:    { checklist:[...] }                                // тактика (redpoint/onsight) —
            //   не дозируется нагрузкой; loadModel:'none'. Находка слоя 3 пула.
```

> **Тактика как `process`.** Redpoint/onsight-тактика — это чек-лист действий, а
> не дозируемая нагрузка. Держим её атомом с `doseShape:'process'` (а не
> отдельной сущностью) ради единого пула и единых гейтов; альтернатива (вынести
> в отдельный тип) — если процессных протоколов станет много и им понадобятся
> свои поля.

Старые поля `workSec/restSec/repsPerSet/setsCount/restBetweenSetsSec` остаются
валидны как `doseShape:'hang'` — обратная совместимость со старыми программами.

> **Находка #5: `intent` (develop/maintain/recover) НЕ в атоме, а на
> блоке/сессии.** Один атом (напр. ARC) в одном цикле — `develop`, в другом —
> `recover`. Поэтому `targetStimulus` живёт в Block (§1.3) и присваивается при
> сборке, не в Exercise.

### 1.3. Блок (Block)

Группа атомов одного качества + параметры дозы и теги для валидатора:

```
Block {
  id, name
  quality                 // главное качество блока
  exercises[]             // атомы
  targetStimulus          // 'develop'|'maintain'|'recover'
  fatigueCost             // нагрузка для модели свежести: 'low'|'moderate'|'high'|'max'
  tissueLoad              // отдельная ось нагрузки на пальцы/ткань (методология 1.6/1.8)
  prerequisites[]         // напр. ['warmup_done','base_>=1y']
}
```

### 1.4. Сессия (Session)

```
Session {
  date
  discipline              // 'boulder'|'sport'|'multipitch'|'general'
  blocks[]                // обычно: warmup → 1–2 primary → secondary → antagonist → cooldown
  readinessGate           // результат readiness.assess(today, history)
  totalFatigue, totalTissueLoad   // агрегаты для валидатора
}
```

Правило порядка блоков (🟢 A): **навык/мощность/сила — на свежесть, в начале;
выносливость и накачка — в конце; антагонисты/мобильность — в самом конце.**
Нельзя ставить технику или max-силу после изматывающей power-endurance.

### 1.5. Циклы

```
Microcycle (week) {
  sessions[]
  rule: ≤2 high-tissue пальцевых сессий, между ними ≥48 ч
  rule: ≥1 день полного отдыха или активного восстановления
}
Mesocycle {
  weeks[]                 // 3–4 нагрузочные + 1 deload (объём −40..60%)
  focus                   // ведущее качество (для линейной модели)
}
Macrocycle {
  model                   // 'linear'|'nonlinear'|'dup'  (методология ч.6)
  goalDate, goalDiscipline
  mesos[]
}
```

### 1.6. Правила сборки (валидаторы)

Конструктор не просто хранит — он **запрещает невалидные сборки**. Минимальный
набор правил-инвариантов (источник требований — методология ч.9, S1–S8):

1. **Возрастной/уровневый гейт** (🟢 A, safety): атом с `minLevel`/`minAge` выше
   профиля — недоступен. Fail-closed (нет данных возраста → блок). Расширяет
   существующий `ageGate`.
2. **Свежесть пальцев** (методология 1.6): нельзя поставить блок
   `tissueLoad: high/max`, если в окне 48 ч (`high`) или 72 ч (`max`) уже была
   `high/max` нагрузка той же `gripGroup` (читается из `fingerTissueLog` /
   history; legacy-история без `gripGroup` трактуется консервативно).
3. **Порядок качеств в сессии** (§1.4): навык/сила/мощность не после
   выносливости.
4. **Прогрессия нагрузки** (методология 1.2/1.8): при сборке следующей недели
   рост high-intensity пальцевого объёма ограничен (~10%/нед, мягкое
   предупреждение).
5. **Разминка обязательна** (методология 5.7): сессия с интенсивными блоками без
   warmup — невалидна.
6. **Баланс антагонистов** (методология 1.8): если за микроцикл нет ни одного
   блока `quality: antagonist` при наличии тяжёлых тяговых — предупреждение.
7. **Deload** (методология 6.4): мезоцикл без разгрузочной недели —
   предупреждение.
8. **Лимитер-приоритет** (методология 1.4): по результатам теста конструктор
   поднимает в рекомендациях блоки на слабое качество.

### 1.7. Как конструктор генерирует тренировку (поток)

```
1. Профиль: уровень, тренировочный возраст, возраст, оборудование, дисциплина-цель.
2. Тест (методология ч.8) → профиль качеств → лимитер.
3. Выбор модели периодизации (методология 6.4) → каркас макро/мезо.
4. На каждую сессию: выбрать качество дня (по модели) → подобрать блоки
   (гейты §1.6) → задать дозу по протоколам (методология ч.5), интенсивность в % от теста.
5. Прогон валидаторов (§1.6). Конфликты → пересборка/предупреждение.
6. Лог выполнения → records → обновление readiness/свежести → вход следующей сессии.
```

## Часть 2. Привязка к HEYS (модель данных)

Существующий модуль `apps/web/fingers/` уже содержит нужный скелет — его не
ломаем, а **обобщаем с «фингерборд» до «любая модальность»**.

### 2.1. Что переиспользуется как есть

- `GRIPS` / `grips_catalog` — биомеханика, `a2ForceRatio`, `dangerLevel`,
  `minAge`, `sourceIds`. Остаётся справочником хватов для пальцевых атомов.
- `bibliography` (`sourceIds`) — механизм привязки к источникам. Распространить
  на все атомы (методология ч.10 даёт стартовый набор источников).
- `readiness.assess(today, history)` — становится входом валидатора свежести
  (§1.6 правило 2).
- `age_gating` (`ageGate.filterPrograms/filterGrips/warnLevel`) — обобщить до
  `filterExercises/filterBlocks` по `minLevel` + `minAge`.
- `records_store`, `calibration`, `timer`, `session_persistence`, `calendar`,
  `safety`, `warmup_runner` — переиспользуются: калибровка ↔ тест (методология
  ч.8), warmup_runner ↔ правило 5; safety ↔ валидаторы.
- `constructor` / `mix_engine` — ядро сборки; расширяется правилами §1.6.

### 2.2. Что обобщить (переименования полей)

| Сейчас (fingerboard-only)                      | Обобщённо                                                          |
| ---------------------------------------------- | ------------------------------------------------------------------ |
| `exercises[i].hangSec`                         | `workSec`                                                          |
| `addedWeightKg` (единственная модель нагрузки) | `loadModel` + `loadValue` (вес / %макс / RPE / grade / bodyweight) |
| программа = набор пальцевых упражнений         | `Block` с `quality` + `modality`                                   |
| `intensity: moderate/max`                      | `quality` + `energySystem` + `fatigueCost` + `tissueLoad`          |

Поля делаются **аддитивно** (новые опциональные рядом со старыми) — старые
программы (`beastmaker_1000_beginner`, `horst_max_hangs`, `lopez_mr` …) остаются
валидными как `modality:'fingerboard', quality:'finger_strength'`.

### 2.3. Новые сущности (новые каталоги-модули)

По паттерну существующих IIFE-модулей `HEYS.Fingers.*` (или новый namespace
`HEYS.Training.*`, если режим выходит за пальцы):

- `quality_catalog` — 9 качеств (методология ч.2) как справочник осей.
- `block_catalog` — каталог блоков (методология ч.4) по модальностям.
- `periodization_engine` — три модели (методология ч.6) как генераторы
  макро/мезо.
- `validators` — правила §1.6.
- `assessment` — батарея тестов и бенчмарки (методология ч.8).

### 2.4. Что НЕ делать при интеграции (из CLAUDE.md проекта)

- Не редактировать `heys_fingers_bundle_v1.js` руками (auto-generated).
- Новые модули — через `scripts/bundle-fingers.cjs` MODULES в нужном layer.
- Safety-критичные правила (гейты возраста/уровня/свежести) — с
  regression-тестами в `TESTS/`.
- Age-gating — fail-closed, как сейчас.

## Часть 3. Алгоритмы Фазы 0 (формулы, снимающие блокеры READINESS)

Здесь — конкретные определения, которых не хватало для кода (см.
`IMPLEMENTATION_READINESS.md`). Решения помечены уровнем: 🟠 — прагматичная
эвристика управления нагрузкой (не валидированная единица), 🟡 — обоснованная
экстраполяция. Они снимают вопросы пула `IMPLEMENTATION_MAP.md`.

### 3.1. Метрика объёма пальцевой нагрузки — FTL (решает `Q-1.2-1`)

**Зачем:** единое число «сколько нагрузили пальцы» за сессию/неделю — вход для
S4 (рост ≤10%/нед), deload (−40..60%), maintenance, границ MEV/MAV (1.7).

**Определение.** Для каждого атома сессии:

```
FTL_atom = TUT_sec × intensityW × tissueW

TUT_sec  — суммарное «время под нагрузкой», по doseShape:
  hang:        workSec × reps × sets
  attempts:    avg(movesPerAttempt) × 2.5с × attempts      // 2.5с ≈ контакт/движение
  circuit:     problemsPerRound × rounds × avgMoveSec(≈25с) // непрерывная связка
  continuous:  workSec × sets                               // ARC/mileage
  reps:        avg(reps) × 3с × sets                        // изометрия/антагонисты
  process:     0                                            // тактика — не нагрузка ткани

intensityW — из loadModel/loadValue, в шкалу ~[0.3 .. 1.5]:
  rm_margin:   1.5 − 0.1×margin   (margin меньше → ближе к отказу → выше)
  pctMax:      pctMax/100
  addedWeightKg: 1 + добавка/вес_тела            // от теста
  bodyweight:  1.0
  rpe:         rpe/7
  grade:       {easy-mid:0.5, submax:0.8, near-limit:1.0, limit:1.2, limit-skill:0.6}
  none:        0

tissueW — из tissueLoad атома: {low:0.3, moderate:0.6, high:1.0, max:1.3}
```

`FTL_session = Σ FTL_atom`; `FTL_week = Σ FTL_session`.

**Применение:**

- **S4:** `FTL_week ≤ 1.10 × trailingAvg(FTL_week, 3–4 нед)` → иначе warn.
- **deload:** целевой `FTL_week = 0.4–0.6 × рабочего`.
- **maintenance:** держать `FTL_week` низким при сохранённой `intensityW` (6.6).
- **MEV/MAV:** полосы `FTL` по качеству/уровню (числа — `Q-1.7-1`).

**Честно (🟠):** валидированной «единицы объёма пальцев» в литературе нет; FTL —
управляемый прокси под принцип ≤10%/нед (1.8), а не доза. Все коэффициенты —
тюнируемые константы конфига, не «истина»; ACWR отвергнут (8.5) в пользу этой
более простой и прозрачной меры. Считается только по `tissueLoad≠low` для целей
травмо-капа; полная сумма — для трекинга.

> Остаются связанные числа: `Q-1.7-1` (полосы MEV/MAV).

### 3.2. Scoring лимитера (решает `Q-1.4-1`, `Q-1.4-3`; структура — числа в `Q-8-1`)

**Зачем:** ядро индивидуализации (1.4 / 8.3) — из теста + навыковых флагов
получить **ведущий лимитер + веса приоритетов блоков** для генератора.

**Алгоритм** (на качество `q`):

```
1. Измеримые q (finger_strength, max_strength, power, anaerobic_capacity,
   aerobic_base, mobility):
     deficit_q = clamp( (benchmark(q, level) − score_q) / benchmark(q, level), 0, 1 )
     // выше 0 = ниже нормы уровня = кандидат в лимитеры; benchmark — таблицы Q-8-1
   ⚠️ Для q БЕЗ уровневого бенчмарка (CF/aerobic_base, W′/anaerobic_capacity —
   §3.5: нормативов нет) deficit_q НЕ вычислим на первом тесте → берём дельту к
   личному ретесту; на первом тесте deficit=0 (нейтрально), лимитер по этим осям
   ловится только навыковым флагом «забивается на длинном» (8.3) до накопления
   ретестов.
2. Неизмеримые тестом q (technique, mental):
     flag_q = (число отмеченных навыковых/тактических маркеров 8.3) / (макс);
     для mental дополнительно — валидные шкалы CSES/CAS-20 (если собраны): нормируем
     к flag_q (метрика приоритетнее чек-листа). [I5/I6, метрики — блок I методологии]
3. Приор уровня (Lattice 8.2): levelPrior_q
     низкий уровень → выше вес technique/mental; элита → выше finger_strength
4. limiterScore_q = max(deficit_q, flag_q) × levelPrior_q
5. ведущий лимитер = argmax(limiterScore);
   веса приоритетов блоков = normalize(limiterScore по q)
6. Переоценка каждый мезоцикл (лимитер мигрирует, 1.4).
```

**Подача в генератор (`Q-1.4-3` — решение: гибрид):** ведущий лимитер задаёт
**фокус мезоцикла** (develop), остальные качества идут как поддержка
(`maintain`) пропорционально весам — не «всё в одно» и не «размазать поровну».

**Честно:** структура стандартная (отклонение-от-нормы × приор уровня) и
обоснованная 🟡; но **числа бенчмарков (`benchmark(q,level)`) и маппинг флагов —
не валидированы**, тюнятся (`Q-8-1` остаётся открытым на ДАННЫЕ, не на
структуру). Маркеры 8.3 собираются по `Q-1.4-2` (продуктовое — как именно: само-
оценка/коуч/видео).

### 3.3. История нагрузки на ткань (решает `Q-1.6-1`, `Q-1.6-2`)

**Зачем:** валидатор свежести S2 и модель восстановления (1.6) требуют, чтобы
знать недавнюю нагрузку **по группам пальцев/ткани**, а не только session-level.

**Решение:** на завершении сессии писать `fingerTissueLog` — записи
`{ dateKey, gripGroup, ftl, tissueLoad, intensityW }` (агрегат из FTL-атомов
§3.1 по группе хвата). **Client-scoped** (инвариант HEYS #9 — ключ под
`currentClientId`, фильтровать foreign-scoped; переиспользовать слой
`records`/dayv2, не глобальный LS).

**Чтение (S2 / readiness):** окно последних N часов по `gripGroup`:

- `Q-1.6-2` — **решение:** окно зависит от `tissueLoad`: **48 ч** для `high`,
  **72 ч** для `max`; ≥1 такой записи в окне → блок повторной high/max нагрузки
  на ту же группу (warn/невалидно). `moderate` — мягкий сигнал, не блок.

**Честно:** 48/72 ч — экспертный консенсус (🟡), параметр конфига; `gripGroup`
маппинг (какие хваты в одну группу по нагрузке на A2) брать из `grips_catalog`.

### 3.4. Контракт machine-load пула (решает `Q-5-2`; Фаза 0)

Чтобы `PROTOCOL_POOL.md` грузился кодом без ручной доработки:

**doseConfidence — поле.** Канон — явный ключ `doseConfidence: 'A'|'B'|'C'` на
атоме; трейлинг-комментарий `# 🟢/🟡/🟠` — источник (loader: 🟢→A, 🟡→B, 🟠→C).
При первом импорте поле проставляется из комментария.

**Sentinel-семантика (зависит от `loadModel`):**

```
loadValue:null + addedWeightKg → «от персонального теста» (вычислить из assessment)
loadValue:null + bodyweight    → «нет добавочной нагрузки»
loadModel:'none'               → недозируемо (process/тактика), loadValue игнор
dose.workSec:0                 → «до отказа» (AMRAP) для данной формы
minAge:null                    → НЕТ ДАННЫХ → fail-closed (блок)
minAge:0                       → явно «без возрастного гейта»
grade loadValue                → enum {easy-mid|submax|near-limit|limit|limit-skill}
```

**`prerequisites_catalog` (токен → как проверяется):**

```
warmup_done        → session.warmupCompleted === true
base_>=Ny          → profile.trainingAgeMonths >= N*12   (парсер N из 1y/2y)
strength_base      → profile.level ∈ {intermediate,advanced,elite} ИЛИ тест силы ≥ порога
injury_screen      → нет активного injury-флага / пройден скрин
safe_fall_setup    → подтверждён сетап страховки/матов (среда)
bfr_cuff_technique → пользователь подтвердил технику манжеты BFR
```

Неизвестный prerequisite-токен → fail-closed (блок атома) + лог. Это завершает
Фазу 0: пул становится machine-loadable, схема §1.2 — полной.

### 3.6. Трансфер-функции «тест → доза» (решает `Q-5-1`)

| Функция                                      | Значение                                                                        |    Дов.     |

### 3.7. Объём: MEV/MAV силы пальцев (решает `Q-1.7-1`)

> Прямого MEV/MAV для пальцев в литературе **НЕТ** — выведено из эффективных
> RCT-доз + правил восстановления. Концепции MEV→MAV→MRV берём у Israetel/RP, но

> медленная соединительная ткань, без болевого предупреждения). 🟡/🟠.

| Параметр                        | Значение                                                                                       | Источник                |            Дов.            |
| ------------------------------- | ---------------------------------------------------------------------------------------------- | ----------------------- | :------------------------: |
| MEV силы (минимум для прироста) | **2 сессии/нед × 3–5 рабочих подходов** макс-висов; ~30–100 с под нагрузкой/нед                | López 2019, Mundry 2021 | 🟢 (доза) / 🟡 (как «MEV») |
| MAV (рабочий объём)             | те же 3–5 подходов × 2(–3) сессии; ёмкость (IntHangs) — 120–250 с/трен                         | López 2019              |             🟡             |

### 3.8. Доли «навык vs железо» по уровням (решает `Q-1.5-1`)

Твёрдая опора (🟡, Lattice n=901): предиктивность силы пальцев **падает с
уровнем** — R² грейда: Combined 49.6% → Advanced(V4–8) 16.7% → Elite(V9–12) 9.9%
→ Higher-Elite(V13+) 4.2%. Power-endurance и гибкость незначимы на всех уровнях.

Доли времени — **экспертный синтез (🟡/🟠), не измерение** (исследования
«оптимальный % на технику» нет):

| Уровень             | Навык/лазание | ОФП+сила пальцев | Прим.                                  |
| ------------------- | :-----------: | :--------------: | -------------------------------------- |
| Новичок (≤V3)       |    80–90%     |      10–20%      | без целевого фингерборда               |
| Средний (V4–8)      |    60–70%     |      30–40%      | фингерборд впервые оправдан            |
| Продвинутый (V9–12) |    55–65%     |      35–45%      | пик специфичной силы                   |
| Элита (V13+)        |    70–80%     |      20–30%      | силу поддерживать, не наращивать валом |

**Старт фингерборда:** 6–12 мес базы / стабильно V3–V4 без боли в пальцах;
главный сигнал — **безболезненное лазание** (🟡).

## Часть 4. SPORT_CONFIG — слой переносимости (sport-agnostic движок)

> **Зачем (из portability-аудита).** Движок (валидаторы, FTL, scoring,
> danger-budget) должен быть **sport-agnostic**, а всё спорт-специфичное (числа,
> оси, риск-ткань) — данными в одном конфиге. Сейчас климбинг-числа §3 и
> danger-budget вшиты в прозу/код — это и есть инстанс `SPORT_CONFIG` для
> скалолазания. При форке на другой спорт **меняется только конфиг**, не движок.

**Контракт `SPORT_CONFIG`** (что выносится из движка в данные):

```
SPORT_CONFIG {
  sportId                 // 'climbing-fingers' | 'armwrestling' | ...
  qualityAxes[]           // оси-качества: {id, energySystem-связь, defaultFatigueCost}
                          //   климбинг: finger_strength, anaerobic_capacity, aerobic_base…
                          //   армрестлинг: wrist_strength, pronation, side_pressure, back_pressure…
  modalities[]            // снаряды: климбинг fingerboard/board/wall/campus; армр. table/band/cable/handle
  positionAxes[]          // ОБОБЩЕНИЕ gripId/edgeSizeMm: массив дименсий позиции
                          //   климбинг: [{gripId}, {edgeMm}]; армр.: [{wristPos}, {elbowAngle}, {technique}]
  tissueRiskModel {       // ⚠️ НЕСУЩЕЕ. Заменяется ПОЛНОСТЬЮ per-sport.
    riskTissue            //   климбинг: A2/A4 pulley; армр.: дистальный bicep tendon / медиальн. эпикондил. / плечо
    loadRatioField        //   климбинг: a2ForceRatio (по хвату); армр.: свой коэффициент по позиции
    dangerBudgetCap       //   кап суммы риск-нагрузки на сессию по bucket
  }
  recoveryWindows         // tissueLoad-tier → часы (климбинг 48/72; per-sport своё)
  intensityWeightMap      // loadModel → intensityW для FTL (§3.1); grade-ветка — климбинг-only
  tissueWeights           // tissueLoad-tier → вес FTL (§3.1)
  benchmarks[level][q]    // §3.5 — целиком per-sport (climbing %BW/20мм здесь)
  transferFunctions       // §3.6 «тест→доза» — целиком per-sport (rm_margin/CF — климбинг)
  mevMav[q][level]        // §3.7 полосы объёма — per-sport
  skillAllocation[level]  // §3.8 доли навык/железо — per-sport
  prerequisitesCatalog    // §3.4 токены→предикаты (climbing: safe_fall_setup, bfr_cuff_technique)
  painLocations[]         // §Q-9-1 (climbing: finger/joint/elbow/shoulder)
  ageGate                 // пороги minAge по риск-ткани (climbing: эпифизы фаланг)
}
```

**Generic (в движке, НЕ в конфиге):** схема атома (§1.2 поля), формы дозы
`doseShape` (`hang`/`iso_hold`, `reps`, `circuit`, `continuous`, `process`), две
оси доказательности, иерархия блок→сессия→циклы и правило порядка, структура
валидаторов S3/S4/S6/S7, формула-каркас FTL (§3.1) и scoring (§3.2),
`loadModel`-enum.

**Инвариант форка:** движок читает спорт-числа ТОЛЬКО из `SPORT_CONFIG`. Литерал
климбинга в коде движка (grip/edge/A2/crimp/CF/hang) = баг переносимости. См.
`PLAYBOOK` → FORK_CHECKLIST.
