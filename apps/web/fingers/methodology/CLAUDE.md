# Методология тренировок для скалолазания (HEYS) — правила проекта

Рабочая папка для разработки методологии режима тренировки в HEYS.

- [`PLAYBOOK.md`](PLAYBOOK.md) — **методичка тиражирования**: переносимый
  процесс «идея → релиз» для будущих спорт-модулей
  (бег/армрестлинг/шея/силовая…). Инварианты, артефакты, анти-паттерны, что 🔁
  переносимо / 🆕 заново.

## Документы

- [`METHODOLOGY.md`](METHODOLOGY.md) — методология (часть 0 сводка школ + части
  1–9 + источники). Активная рабочая зона.
- [`CONSTRUCTOR_SPEC.md`](CONSTRUCTOR_SPEC.md) — слой реализации (конструктор +
  привязка к `HEYS.Fingers.*`). Размораживается по мере наполнения пула.
- [`PROTOCOL_POOL.md`](PROTOCOL_POOL.md) — канонический пул протоколов (атомы),
  размечен по осям §5.0. `sourceIds` — только реальные ID из реестра, без
  `generic_*`.
- [`BIBLIOGRAPHY.md`](BIBLIOGRAPHY.md) — машиночитаемый реестр источников
  (`sourceId` → полная ссылка + уровень). Новую ссылку заводить сюда ДО
  использования в `sourceIds`.
- [`AUDIT_SOURCES.md`](AUDIT_SOURCES.md) — **технический архив**: слои аудита
  источников (§1–6) + журнал внедрения (§7). Сюда фиксировать новые находки/
  бэклог, НЕ в методологию.
- [`IMPLEMENTATION_READINESS.md`](IMPLEMENTATION_READINESS.md) — аудит
  готовности к коду (данные/алгоритмы/интеграция) + фазовый план Фаза 0→1→2.
- [`IMPLEMENTATION_MAP.md`](IMPLEMENTATION_MAP.md) — трассировка «единица
  методологии → код» по стабильным ID (анти-потеря: пустая ячейка видна сразу).
- [`tools/impl-coverage.mjs`](tools/impl-coverage.mjs) — чекер покрытия карты.

## Правило: карта реализации покрывает методологию

`IMPLEMENTATION_MAP.md` — ключ-таблица по стабильным ID (подразделы `N.M`, блоки
`block-X`, валидаторы `Sk`, `part-N`). Содержание методологии НЕ дублируется —
только трассируется. **Синхронизация — по покрытию, не по контенту:** после
добавления/переименования единицы методологии прогнать
`node tools/impl-coverage.mjs` и завести/поправить строку в карте (чекер покажет
пропуски/орфанов). Это часть процедуры «правка завершена» наряду с
`school-weights.mjs`.

**Пул открытых вопросов.** Все «вопросы для уточнения» из карточек собираются в
единый реестр `# Пул открытых вопросов` (ID `Q-<единица>-<n>`, статус 🔵 open →
✅ решено + запись решения). Канон — пул; в карточке вопрос дублируется коротко
со ссылкой на ID. Чекер сверяет, что каждый `Q-*` есть и в карточке, и в пуле.
По мере развития плана решённые вопросы помечать ✅ и фиксировать решение.

- [`tools/school-weights.mjs`](tools/school-weights.mjs) — скрипт пересчёта
  сводной таблицы школ (часть 0).

## Правило: сводка школ всегда актуальна

Часть 0 `METHODOLOGY.md` содержит авто-таблицу «доли опоры» школ между маркерами
`SCHOOL-WEIGHTS:START/END`. Она НЕ правится руками.

**После любой правки содержания методологии** (добавил/убрал протокол, источник,
сместил акценты) — обязательная процедура «правка завершена»:

1. **Пересчитать проценты и акценты:** прогнать `node tools/school-weights.mjs`.
   Таблица в части 0 перегенерируется из актуального текста.
2. **Поддержать состав строк динамически:**
   - Если правкой введён **новый ощутимый аспект/школа/течение** (заметная
     опора, не разовое упоминание) — добавить для него паттерн в массив
     `SCHOOLS` скрипта, чтобы он появился отдельной строкой.
   - Если аспект **перестал быть активно отражён** (упоминания ушли в ноль) —
     строка скрывается автоматически (скрипт фильтрует `count == 0`); если школа
     ушла насовсем — убрать её паттерн из `SCHOOLS`.
3. Перепроверить, что сумма процентов = 100 и порядок по убыванию (скрипт делает
   сам через largest-remainder).

Метрика (v3) — **взвешенная с защитой от перекосов**: вклад школы = Σ*раздел [
вес*раздела × √(взвеш. упоминания в разделе) ], где упоминание рядом с
дозировкой/протоколом весит ×1.4 (`PRESCRIPTIVE`). Веса разделов — ядро ×1.5 /
поддержка ×1.0 / источники и сводка ×0 (`PART_WEIGHT`). √ — насыщение: повтор
имени в одном разделе почти не добавляет (опора = охват, не частота). Бонус — за
предписание (что/сколько делать), а не за слова-рамки: это не даёт
«разговорчивым» школам (напр. MacLeod — архитектурный, не предписывающий слой)
раздуваться. Колонка `focus` в `SCHOOLS` — ручная подпись «акцент»: что
конкретно берём у школы; обновляй при правке акцентов. Если меняешь структуру
документа — синхронизируй `PART_WEIGHT`. Порог «ощутимости» новой школы —
экспертное суждение: разовая ссылка строку не заводит, сквозная опора заводит.

## Правило: источник истины по числам + две оси доказательности

- **Канонические числа протоколов — `PROTOCOL_POOL.md`**, не методология. Часть
  5 `METHODOLOGY.md` — слой обоснования (механизм, принцип, диапазоны) + карта
  соответствия 5.8. Меняешь дозу — правишь пул; в части 5 держишь только
  принцип/ диапазон. Не дублировать точные числа в двух местах.
- **Две оси доказательности, не путать:** _сила источника_ (🟢/🟡/🟠 в
  `BIBLIOGRAPHY.md`) ≠ _`doseConfidence`_ (🟢/🟡/🟠 у протокола, доказанность
  конкретных чисел). Протокол на 🟢 A источнике может иметь дозу B
  (экстраполяция).

## Процесс

1. Пользователь даёт правки/вводные и обсуждает части 1–9.
2. После согласования методологии — переход к `CONSTRUCTOR_SPEC.md` (части 10–11
   реализации).
3. Только потом — код в `apps/web/fingers/` репозитория HEYS-v2.

## Главный принцип работы: научность > мнения

**Приоритет — научная обоснованность и логическая состоятельность, а не согласие
с пользователем.**

- Если умозаключение/гипотеза пользователя **расходится с официальной наукой,
  устоявшимися протоколами или признанными программами** — я **сразу прямо
  говорю об этом**, с указанием источника/механизма и уровня доказательности (🟢
  A / 🟡 B / 🟠 C). Не поддакиваю, не подстраиваю текст под мнение в обход
  фактов.
- Несогласие формулирую конструктивно: что именно противоречит, чему именно
  (источник), и какова цена ошибки (особенно если затрагивает безопасность).
- Где наука неоднозначна или данных нет (уровень B/C) — честно говорю, что это
  экспертный консенсус/экстраполяция, и тогда мнение пользователя как практика
  имеет вес. Различаю «противоречит данным» и «не покрыто данными».
- Экспертный опыт пользователя ценен и учитывается — но не может молча
  переопределять прямые данные (уровень A). При конфликте: данные побеждают,
  расхождение проговаривается явно.
- Безопасность (методология ч.9) не обсуждается на понижение: гейты, нагрузка на
  ткани, age-gating — fail-closed.
