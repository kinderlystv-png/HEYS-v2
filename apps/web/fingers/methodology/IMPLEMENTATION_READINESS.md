## Контекст 1 — модель данных пула

**Содержательно чисто:** все enum-значения в допустимых множествах (modality,
quality 9/9, doseShape, loadModel, grade-ярлыки), `dose`-формы соответствуют
`doseShape`, все 20 `sourceIds` и оба `gripId` (`halfcrimp/openhand4`)
резолвятся. Плейсхолдеров нет.

**Но загрузить 1:1 по §1.2 сейчас нельзя** — нужно:

1. **[BLOCKER] `doseConfidence` живёт в комментарии** `# 🟢/🟡/🟠`, а не в поле.
   После парсинга YAML теряется. → вынести в явный ключ `doseConfidence: A|B|C`
   на все 36 атомов (+ 2 антагониста без явной буквы).
2. **[IMPORTANT] Структурное расхождение пул ↔ §1.2:** в пуле
   `gates{...}`-обёртка и `fatigueCost/tissueLoad/prerequisites` на **атоме**, а
   в §1.2 они плоские/на **Block**. → согласовать (рекомендация: принять их на
   атом — пул и §2.2 уже так трактуют).
3. **[IMPORTANT] `prerequisites` — орфаны без словаря:** `warmup_done`,
   `base_>=1y/2y`, `strength_base`, `bfr_cuff_technique`, `injury_screen`,
   `safe_fall_setup`. → завести `prerequisites_catalog` (токен → как
   проверяется) + парсер `base_>=Ny`.
4. **[IMPORTANT] Sentinel-семантика неоднозначна:** `loadValue: null` под
   `addedWeightKg` = «от теста», под `bodyweight` = «нет нагрузки»; `workSec:0`
   = «до отказа»; `minAge:0` = «без гейта» vs `null` = fail-closed. →
   задокументировать по контексту `loadModel`; парсер ветвится.

После этих 4 пунктов пул грузится в `block_catalog`/`assessment` без ручной
доработки. Остальное (квотинг `'base_>=1y'`, name/desc-поля, i18n чек-листов) —
косметика.

## Контекст 2 — алгоритмы

Кодируемо «как есть»: гейты **S1/S3/S6/S7**, порядок качеств, **каркас времени**
(micro/meso/deload/taper/maintenance), **linear-скелет** периодизации,
лог→свежесть (после фикса агрегации). Это ~70% валидаторов и генератор скелета
для линейной модели.

**Три блокера (без них «бей в лимитер» и безопасная доза не алгоритмизуются):**

1. **🔴 Нет метрики «объём пальцевой нагрузки».** Нужна для S4 (рост ≤10%/нед),
   магнитуды deload (−40..60%), maintenance, taper. ACWR явно отвергнут (8.5) —
   другой опоры нет. → определить формулу (напр. `Σ reps×sets×workSec` по атомам
   `tissueLoad∈{high,max}`, с нормировкой разных `doseShape`).
2. **🔴 Нет transfer-функций «тест → доза».** `rm_margin` (главная модель силы
   пальцев) — есть принцип «держал бы +N сек», нет кривой сила-длительность
   тест→вес. Аналогично `grade enum→грейд` (сдвиги), `pctMax/CF→число`
   (конверсия добавочного веса в MVC/CF). → 3–4 формулы/таблицы закрывают всю
   дозировку.
3. **🔴 Нет scoring лимитера (8.3 шаг 3) + бенчмарк-таблиц по уровню (8.2).**
   Методология честно называет это экспертным суждением. Но без функции
   «отклонения тестов + флаги маркеров → один ведущий лимитер/веса блоков» не
   работают §1.7 шаг 2 и валидатор §1.6.8 — то есть ядро индивидуализации.

**Генераторы периодизации:** linear — кодируем; **nonlinear/DUP — только пример,
нет параметрических шаблонов ротации**; нет «бюджета объёма/частоты на
качество». **Moderate:** проводка CF/grade/pctMax в дозу, конфликт-резолюция
§1.7.5, доформализация предикатов («интенсивный блок» S3, «тяжёлая тяга» S6,
агрегация tissueLoad S2, ветка rehab S8).

## Контекст 3 — интеграция с `apps/web/fingers/`

**Реалистично, во многом аддитивно — спека писалась против реального кода.**
~60–70% Части 1 (атом + сессия + валидаторы §1.6) ложится на существующие модули
без слома.

- **`exercises[i]` → `Exercise`:** rename-map §2.2 работает 1:1 как
  `doseShape:'hang'` (hangSec→workSec и т.д.); новые поля (quality/modality/
  energySystem/doseShape/loadModel/doseConfidence) добавляются **опционально, не
  ломая** старые программы.
- **GRIPS catalog — переиспользуем как есть:** `a2ForceRatio` (halfcrimp 4.0,
  fullcrimp 33.0), `dangerLevel`, `minAge`, `sourceIds` — все на месте. ⚠️
  грип-каталог ссылается на `schweizer2008` — наш фикс
  (`schweizer2001`+`moor2009`) надо донести и в код.
- **`mix_engine` уже реализует значительную часть §1.6/§1.7 на уровне ОДНОЙ
  сессии:** danger-budget по `a2ForceRatio`, обязательный антагонист, порядок
  слотов, age+pain+readiness-гейты, %MVC-дозирование с потолком, объём-кап. Это
  фактический «движок сборки» (не `constructor`, который лишь UI-редактор
  атома).
- **Переиспользуем+расширяем:** `ageGate.*` (тонкая `filterExercises`),
  `readiness.assess` (нужна per-grip/tissue история — её нет), `mix_engine`
  (роль→9 качеств). Новые модули §2.3 чисто вписываются в bundle-паттерн.
- **Greenfield (строить заново):** слой циклов (неделя/мезо/макро — выше сессии
  сейчас НЕТ ничего); `doseShape` за пределы hang (attempts/circuit/continuous/
  process) + не-фингерборд модальности; сущность `Block`;
  `periodization_engine`.

**Риски:**

- **🔴 HIGH — safety-регрессия.** mix_engine/ageGate/readiness — сцепленные
  fail-closed гейты, а тест `TESTS/heys_fingers_safety.test.js` покрывает только
  age + вес (НЕ danger-budget, readiness-48ч, pain-gate). → **расширить
  safety-тест ДО любого рефактора.**
- **🔴 HIGH — свежесть ткани.** Недельное правило «≤2 high-tissue/48ч» требует
  per-grip истории нагрузки, которой нет (readiness знает только session-level).
  → новая прокладка через dayv2/fingersLog.
- **🟡 MED — дублирование расчёта длительности** (hang-формула в 2 местах) →
  дрейф при обобщении doseShape.
- **🟡 MED — cross-client:** персистенция планов макро/мезо обязана быть
  client-scoped (инвариант #9 HEYS).
