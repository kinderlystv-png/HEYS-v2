HEYS-v2 — Unified Audio System Master Prompt

Ты работаешь в репозитории HEYS-v2 и должен полностью переработать звуковое и тактильное сопровождение клиента, создав единый централизованный компонент sound/haptic UX для всего проекта.

==================================================
1. ГЛАВНАЯ ЦЕЛЬ
==================================================

Сделать премиальный, консистентный, логичный и уместный уровень звукового сопровождения во всём приложении.

Нужно прийти к результату, в котором:
- есть единый центр управления всеми звуками проекта;
- есть единый центр управления всеми haptic/vibration-эффектами;
- позитивные события звучат приятно, но разнообразно;
- негативные события звучат напряжённо, но не раздражающе;
- разные уровни негативности ощущаются по-разному;
- интерфейсные события имеют деликатный, дорогой по ощущению отклик;
- звук и вибрация работают как единая UX-система;
- итог ощущается как premium / enterprise-grade polish.

==================================================
2. ЧТО НУЖНО ПОЛУЧИТЬ НА ВЫХОДЕ
==================================================

Нужно спроектировать и внедрить новый единый модуль, например HEYS.audio, который:

1) управляет всеми звуками проекта;
2) управляет всеми haptic/vibration-эффектами;
3) имеет единые настройки;
4) обеспечивает backward compatibility со старыми вызовами;
5) покрывает ключевые пользовательские события;
6) остаётся расширяемым и безопасным для legacy runtime;
7) имеет механизм аварийного отката/kill switch на случай regressions в предпродакшне;
8) не создаёт лишнюю нагрузку на critical path загрузки.

Итоговый API должен быть примерно такого уровня:
- HEYS.audio.play(eventName, options?)
- HEYS.audio.playCategory(category, options?)
- HEYS.audio.haptic(typeOrEvent, options?)
- HEYS.audio.getSettings()
- HEYS.audio.setSettings(partial)
- HEYS.audio.testCategory(category)
- HEYS.audio.stopAll?()
- HEYS.audio.canPlay?(eventName)

- `prefers-reduced-motion` media query уже используется в системе; новый модуль обязан его уважать.

==================================================
4. ПРОДУКТОВОЕ ВИДЕНИЕ СИСТЕМЫ
==================================================

Нужно создать единый язык звуков и тактильных ощущений.

4.1. Позитивные события

Позитивные события не должны звучать одинаково.
Нужно выделить несколько приятных семейств:
- мягкое подтверждение;
- награда;
- успех;
- достижение;
- крупный триумф;
- завершение полезного ритуала.

Примеры позитивных событий:
- добавление хорошего продукта;
- добавление воды (water add) — одно из самых частых действий;
- достижение цели по воде (water goal reached);
- получение XP;
- выполнение миссии;
- выполнение всех миссий;
- завершение дневного ритуала добавок;
- достижение цели по калориям;
- level up;
- rank/title ceremony;
- streak milestone;
- завершение refeed-дня в ok-зоне.

4.2. Негативные события

Негативные события должны иметь несколько уровней напряжения.
Не нужен один универсальный warning.

Нужны разные градации:
- мягкое предупреждение;
- настораживающий сигнал;
- тревожный сигнал;
- ошибка/сбой.

Примеры:
- добавлен умеренно вредный продукт;
- добавлен сильно вредный продукт;
- сильный перебор по калориям;
- broken streak;
- ошибка сохранения;
- API error.

Ключевой продуктовый принцип:
Чем хуже событие, тем более напряжённая тональность, но без неприятной сирены и без раздражения при ежедневном использовании.

4.3. Нейтральные и интерфейсные события

Нужны деликатные, очень уместные звуки/вибрации для:
- tab switch;
- toggle;
- expand/collapse;
- modal open/close;
- advice appear/dismiss;
- swipe dismiss;
- pull-to-refresh;
- небольших подтверждений действий.

Важно: не надо делать приложение "болтливым". Звук должен усиливать смысловые моменты, а не сопровождать каждое микродвижение пользователя.

7. ОСОБОЕ ТРЕБОВАНИЕ ПО ВРЕДНЫМ ПРОДУКТАМ
==================================================

Если пользователь добавляет вредный продукт, событие должно сопровождаться:
- напряжённым, но уместным звуком;
- правильной вибрацией;
- дифференциацией по степени негативности.

То есть:
- умеренно плохое -> caution;
- сильно плохое -> alert.

Нужно использовать реальные поля проекта, и помнить о правилах репозитория:
- использовать product.harm, а не product.harmScore.

До начала реализации нужно отдельно зафиксировать продуктовую шкалу:
- какие диапазоны `product.harm` считаются neutral / caution / alert;
- должен ли обычный food add звучать всегда или только для значимых случаев;
- должен ли harmful feedback срабатывать только после confirm-add, а не на preview/hover/intermediate state.

==================================================
8. ТРЕБОВАНИЯ К АРХИТЕКТУРЕ
==================================================

Новая система должна иметь:
- единый AudioContext;
- единый контроль громкости;
- единый master-toggle;
- отдельный haptic toggle;
- fallback на старые интеграции через bridge/wrapper;
- отсутствие регрессий в legacy runtime;
- совместимость с уже существующими модулями;
- систему дедупликации/анти-спама для предотвращения каскада звуков;
- приоритизацию событий, чтобы triumph/error могли перекрывать interaction/noise;
- правила поведения в hidden/background tab;
- graceful degradation для браузеров без AudioContext/vibrate support;
- уважение `prefers-reduced-motion` — при active reduced-motion отключать haptic и анимированные звуковые эффекты;
- единственный AudioContext на весь модуль, переиспользуемый между вызовами;
- корректный suspend/resume lifecycle AudioContext при visibility change;
- защита от утечки памяти: неиспользуемые осциллаторы должны disconnect и garbage-collect.

Нужно сделать backward compatibility:
- старые HEYS.sounds вызовы не должны ломаться;
- HEYS.daySound не должен ломаться;
- локальные игровые sound-функции должны либо мигрировать, либо стать thin wrappers;
- разрозненные настройки нужно свести к единой схеме с миграцией.

Архитектурные guardrails:
- не тащить тяжёлый звукодвижок в boot-core без доказанной необходимости;
- если нужно раннее подключение, рассмотреть thin bootstrap/shim + основную реализацию позже;
- не допускать, чтобы новый модуль ухудшил first-load или инициализацию приложения;
- не плодить параллельно два независимых источника истины по sound settings.

9. ТРЕБОВАНИЯ К НАСТРОЙКАМ
==================================================

На первом этапе нужен простой UI:
- master toggle;
- volume slider;
- haptic on/off.

Не надо перегружать UI десятком подкатегорий.
Но внутренняя архитектура должна быть готова к будущему расширению.

Нужен единый settings key, например:
- heys_audio_settings

Старые настройки нужно учитывать как источник миграции/фоллбэка, но новой точкой истины должен стать единый объект.

Дополнительно:
- нельзя молча сделать систему существенно громче/активнее для пользователей, у которых ранее звук был выключен;
- нужна явная стратегия миграции legacy keys -> new key;
- нужно решить, синхронизируется ли `heys_audio_settings` между устройствами и как избежать конфликтов со старыми ключами.

10. РЕКОМЕНДУЕМЫЙ ПЛАН РАБОТ
==================================================

Фаза 0. Подготовка фундамента и снятие блокеров

Перед началом реализации НЕОБХОДИМО закрыть следующие блокеры и неопределённости:

0.1. Зафиксировать product semantics
- определить пороги для `product.harm`: neutral / caution / alert;
- определить, какие frequent actions действительно заслуживают звук, а какие должны быть только haptic или вообще без отклика;
- определить default policy для everyday interactions: звук, haptic-only или silent.

0.2. Зафиксировать migration contract
- составить old -> new mapping для всех текущих sound/haptic entrypoints;
- зафиксировать, какие прямые `navigator.vibrate(...)` остаются как low-level tactile ticks и НЕ мигрируются в первой волне;
- определить, какие старые ключи читаются только для миграции, а какие продолжают поддерживаться временно.

0.3. Принять решение по bundle placement
- оценить риск размещения полного `HEYS.audio` в `boot-core`;
- если это ухудшает critical path, разделить на bootstrap API и основной engine;
- зафиксировать целевую bundle strategy до начала кодинга.

0.4. Зафиксировать runtime guardrails
- user-gesture unlock для AudioContext;
- правила работы в background/hidden tab;
- cooldown/dedupe window для частых событий;
- поведение при burst-сценариях, когда за короткий интервал прилетает несколько событий.

0.5. Зафиксировать platform support matrix
- iOS Safari / Android Chrome / desktop Chrome / desktop Safari;
- поддержка/ограничения vibrate API;
- поведение в embedded/PWA/webview среде;
- graceful fallback, если audio/haptic недоступны.

0.6. Зафиксировать safety/rollout plan
- добавить kill switch / feature flag для emergency disable новой системы;
- предусмотреть временный fallback на legacy behavior;
- определить минимальный smoke-test matrix до merge.

0.7. Зафиксировать observability/debugability
- добавить debug logging с префиксом `[HEYS.audio]` без персональных данных;
- предусмотреть способ быстро понять: событие не сработало из-за mute/settings/support/cooldown/visibility.

0.8. Зафиксировать anti-overkill boundaries
- не пытаться в первой волне озвучить буквально каждый tap;
- не заменять все low-level tactile ticks без сильной причины;
- не добавлять внешние audio assets в v1, если синтезаторный подход уже покрывает качество и уменьшает bundle-risk.

0.9. Зафиксировать default-on policy
- определить, включается ли новая система по умолчанию для ВСЕХ пользователей или стартует в opt-in;
- для пользователей, у которых ранее звук был ВЫКЛЮЧЕН, новая система НЕ должна включать звук автоматически;
- для новых пользователей определить, какие категории активны out-of-the-box;
- рекомендуемый дефолт: masterEnabled = true, volume = 0.12 (мягче чем мидл), hapticEnabled = true.

0.10. Зафиксировать water-add frequency concerns
- water add — одно из самых частых действий в приложении (до 8-12 раз в день);
- нужно решить до кодинга: water add = haptic-only по дефолту или лёгкий sound;
- если sound, обязателен per-event cooldown, чтобы не надоесть;
- water goal reached — полноценный success event.

0.11. Зафиксировать accessibility contract
- модуль обязан проверять `prefers-reduced-motion` и `prefers-reduced-transparency`;
- при active reduced-motion: отключить haptic, оставить только звук (или и его ослабить);
- при отсутствии поддержки vibrate API: не ломать поведение, не кидать ошибок;
- iOS ringer-switch (silent mode) может заглушить Web Audio — это ОК, не пытаться это обходить.

==================================================
12. ТРЕБОВАНИЯ К SOUND DESIGN
==================================================

Новая система должна звучать дорого и уместно.

Требования:
- не делать всё одинаковыми короткими бипами;
- позитивные категории должны различаться характером;
- негативные категории должны различаться напряжённостью;
- интерфейсные звуки должны быть очень деликатными;
- не перегружать пользователя;
- не превращать приложение в игровую аркаду, но сохранить wow-эффект.

Дополнительные sound-design guardrails:
- короткие, чистые, современно звучащие сигналы лучше длинных навязчивых фраз;
- избегать резких высокочастотных пиков и клиппинга;
- interaction/utility sounds должны быть тише и короче reward/success/triumph;
- repeated actions должны иметь cooldown, чтобы не превращаться в шум;
- система должна быть "музыкальной", но не мультяшной.

То, что уже нравится пользователям, нужно учесть как референс.
Например, приятный успех-звук можно сохранить как основу одной из категорий, но не использовать его для всего подряд.

==================================================
13. ТРЕБОВАНИЯ К HAPTIC DESIGN
==================================================

Вибрация — не вторичный слой, а часть общей системы.

Нужно:
- связать haptic с sound categories;
- использовать лёгкие паттерны для interaction;
- использовать приятные паттерны для success/reward;
- использовать более заметные паттерны для caution/alert/error;
- сохранить ощущение премиального продукта, а не хаотичных вибросигналов.

Дополнительные haptic guardrails:
- не дублировать агрессивную вибрацию на каждое частое действие;
- drag/slider/picker micro-ticks мигрировать только если это улучшает консистентность без потери отзывчивости;
- если на платформе haptic недоступен, звук и UI не должны предполагать, что вибрация состоялась.

14. ВАЖНЫЕ ПРАВИЛА РЕПОЗИТОРИЯ

- ответы и комментарии — по-русски;
- код — на английском;
- не делать rollback-команды без явного запроса;
- не использовать localhost API вместо production API;
- для legacy runtime в apps/web/** rebuild обязателен;

- для runtime-sensitive изменений нужна browser verification;
- использовать project conventions и существующие HEYS.* паттерны;
- не оставлять `console.log`, только `console.info/warn/error` с префиксом `[HEYS.audio]`;
- не логировать персональные данные;
- помнить, что `localhost` читает реальные public bundle assets, а не только исходники.

15. КРИТЕРИИ ГОТОВНОСТИ
==================================================

Задача считается выполненной, если:

1. появился единый sound/haptic module;
2. старые модули не ломаются и работают через новую систему либо совместимы с ней;
3. есть единые настройки;
4. покрыты ключевые пользовательские события;
5. вредные продукты имеют адекватно более напряжённый отклик;
6. success / reward / triumph реально различаются между собой;
7. haptic интегрирован как часть общей системы;
8. выполнен rebuild нужных bundles;
9. выполнена post-build validation;
10. выполнена browser verification, если требуется по риску изменений;
11. есть Phase 0 artefacts: решение по bundle placement, migration contract, thresholds и kill switch strategy;
12. система не деградирует UX в burst/frequent interactions;
13. зафиксированы сознательно отложенные low-level точки, если они есть.

==================================================
17. ЭМОЦИОНАЛЬНЫЙ ДИЗАЙН И СОВРЕМЕННЫЕ ИДЕИ
==================================================

Чтобы система звучала по-настоящему современно и вау, стоит учитывать:

17.1. Audio language, не набор бипов
Все звуки должны принадлежать одному музыкальному семейству.
Например, одна тональность / гамма, один характер тембра — чтобы triumph и reward ощущались как родственники, а не случайные звуки из разных приложений.

17.2. Emotional arc через день
При длительном ежедневном использовании пользователь проходит "эмоциональную дугу":
- утро: первый лог, первое добавление — сдержанно, мягко;
- день: активная фаза — ритмичные reward'ы;
- вечер: итоги, goal reached, успех — кульминация.
Система пока не обязана менять звуки по времени суток, но архитектурно event -> category -> synthesis chain позволяет это в будущем.

17.3. Haptic-first для interaction layer
Современный подход для UI-фидбека:
- toggle, tab, dismiss, expand/collapse — в основном haptic;
- звуки для interaction только в редких, значимых случаях;
- sound = семантические моменты (результат, награда, предупреждение);
- haptic = физическое ощущение касания.

17.4. Signature sound
Один "фирменный" звук, который пользователь будет ассоциировать с HEYS.
Лучший кандидат — звук при достижении цели / success. Его можно сохранить как brand-cue и использовать только для ключевых побед.

17.5. Sparse > Frequent
Премиум-ощущение рождается из:
- редкости (не каждый тап);
- точности (именно в нужный момент);
- различимости (каждый звук несёт уникальный смысл);
- контраста (тишина делает звук ценнее).

17.6. Progressive richness (идея на будущее)
Звуки могут эволюционировать:
- streak 1 day: обычный success;
- streak 7 days: success с лёгким бонусным тоном;
- streak 30 days: success с дополнительной нотой;
Это не обязательно в первой волне, но архитектура event -> options -> synthesis позволяет.

17.7. Rarity-based intensity (идея на будущее)
Для gamification:
- common достижение = простой chime;
- rare = более богатый;
- legendary = мини-фанфара;
Это тоже не обязательно сейчас, но event mapping позволяет.

==================================================
18. КОРОТКАЯ СУТЬ ДЛЯ ИСПОЛНИТЕЛЯ
==================================================

Сделай не просто набор звуков, а единую премиальную систему звукового и тактильного сопровождения для HEYS-v2:
- консистентную;
- архитектурно чистую;
- приятную в позитивных сценариях;
- напряжённую, но уместную в негативных сценариях;
- масштабируемую;
- совместимую с текущим runtime;
- действительно ощущающуюся как high-quality client experience;
- sparse, а не chatty — сила в точности, не в количестве;
- музыкально целостную — одно семейство звуков, а не коллаж;
- accessibility-aware — уважает prefers-reduced-motion и silent mode.
