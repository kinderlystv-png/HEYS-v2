# Маркетинг HEYS — пакет материалов

Рабочий пакет по продвижению HEYS на рынке РФ. Подготовлен 2026-06-09,
актуализируется под Pro-first запуск. Аудитория: основатель / команда
(внутренний рабочий план). В пакете есть активные источники, runbooks и архивные
исследования; не каждый numbered-файл является живым источником правды.

## Как читать

1. Открыть `00_Дашборд.html` — текущая картина без чтения всего корпуса.
2. Открыть `22` — взять живые задачи, статусы и release-ступени.
3. Открыть `25` — проверить gates и красные триггеры перед запуском/масштабом.
4. Открыть `15` — понять, почему решения приняты, и добавить новые решения.
5. По зоне работы открыть runbook: Telegram `14`/`24`, ПДн/RKN `31`/`32`, ERID
   `33`, куратор `23`, измеримость `18`.
6. В canonical/evidence документы `01`-`04`, `06`-`13`, `16`, `19`-`21`, `26`,
   `29`, `30`, `34`-`38` ходить только за каноном/обоснованием, а не за живыми
   статусами.

## Приоритеты NOW

1. **Релизный фокус 2026-06-12: три блока** — landing perfect-pass по
   конкурентке (`30` L2/L3/L6), продуктовый shortlist через задачи `22`,
   Telegram как первый управляемый канал (`14`/`24`).
2. **Единая концепция позиционирования** — HEYS продаёт не счётчик и не диету, а
   внимательную внешнюю опору: куратор ведёт дневник, держит контекст недели,
   помогает увидеть паттерны и вернуться без стыда. Мягкая внешняя
   ответственность — механизм этой опоры; анти-срыв — сильная entry-боль, но не
   вся категория продукта (`03`, `14`, `23`, `COPY_VOICE`).
3. **Запуск Фазы 0 = Self / Pro / Pro+** — Self 490 ₽ как самостоятельный
   урезанный режим; Pro 7 990 ₽ для первого набора, 7 дней только после
   квалификации; Pro+ индивидуально (`19`). Lite — открытый вопрос, Solo AI —
   позже.
4. **AI-распознавание** — не обещать и не внедрять в Pro; ручной ввод куратора
   нужен для внимания, опыта и будущей методологии (`20`).
5. **Позиция по данным** — вынести в лендинг/FAQ: HEYS не заменяет точность
   полуработающей автоматизацией (`21`).
6. **Юр-структура кураторов (НПД)** — не блокирует первые 10–15 клиентов, пока
   основатель сам куратор. Договор на результат нужен перед первым внешним
   куратором / Ф1 (`22` 4.9), не перед стартом Ф0.
7. **ПДн/RKN release gate** — локальные R0-артефакты собраны, РКН updateform по
   HEYS принят порталом 2026-06-17, но gate остаётся 🟡: до активного сбора
   заявок и триалов закрыть `22` 1.7–1.9 — публикация/сверка РКН-записи, защита
   ИСПДн и R0 из `32`. До первых оплат — R1 по ЮKassa/чекам/events; до
   paid-scale/Ф1 — R2/R3 с legal sign-off.

   | Подпункт            | Статус | Следующее действие                                                    |
   | ------------------- | ------ | --------------------------------------------------------------------- |
   | `22` 1.7 РКН        | 🟡     | дождаться публикации/сверки после принятого updateform                |
   | `22` 1.8 ИСПДн      | 🟡     | финальный уровень ИСПДн, модель угроз, Telegram/payment smoke         |
   | `22` 1.9 governance | 🟡     | публикация РКН, импорт `.ics`, первая monthly запись, drill, sign-off |
   | `32` R1 платежи     | ⬜     | проверка ЮKassa/payment metadata перед оплатами                       |
   | `32` R2/R3 масштаб  | ⬜     | legal sign-off, incident drill, договоры внешних кураторов            |

8. **Оплата Pro** — ручная помесячная; продление предлагает куратор, когда видит
   позитивный момент и понимает контекст клиента.
9. **Измеримость при отключённой аналитике** — DB-воронка как источник правды,
   промокоды + dropdown «откуда узнали»; серверная Metrica — до платного
   масштабирования.
10. **ERID-маркировка** — до первой закупки платной рекламы закрыта в `33`:
    владелец — основатель, посев без ERID-строки/договора/отчётной даты не
    публикуется.

> **Про «блокер W1» (экономика куратора).** На Фазе 0 он не действует в полном
> виде: первый набор идёт на внутреннем ресурсе, без внешней выплаты куратору по
> полной ставке. Блокер превращается в задачу Фазы 1: на сколько поднять цену
> под зарплату внешнего куратора (лист «Фазы-цена», ориентир ~9 900–10 900 ₽).
> Старая цифра «фикс 7 500» из брифа — под устаревшую цену 12 990.

## Важная оговорка по цифрам

Цены и операционные параметры (тарифы, загрузка кураторов) — из проекта
(`pricing.ts`, `docs/HEYS_BRIEF.md`). Маркетинговые допущения (CAC, конверсии,

ДОПУЩЕНИЯ. Их нужно заменить фактом по мере накопления данных за Фазу 1–2.

## Ритм актуализации (cadence)

Чтобы корпус не устаревал и не расползался — лёгкий еженедельный ритуал, **≤
15–30 мин**:

1. **KPI-факт** — свежие цифры в лист «KPI-трекер» `00_Сводная_панель.xlsx`;
   проверить красные метрики.
2. **Статусы** — обновить `22_План` и `25_Roadmap` (что сдвинулось, что
   красное). `30` статусы тянет из `22` сам — руками не править.
3. **Решения** — договорённости недели → в журнал `15_Ревизия` §2 (а не в чат).
4. **Datestamp-проверки** — освежить при устаревании: конкурентные цены (`29`),
   закон о рекламе нутрициологии (`10`/`22` п.2.5), ПДн/RKN gate (`31`/`32`).
5. **ПДн-сверка** — раз в месяц проверить код/БД/лендинг/документы/РКН/кураторов
   по `32`; gaps открыть задачами в `22`, решения — в `15`.

Правила:

- **Триггер changelog:** правка ключевого дока > 10% или изменение канона →
  запись в Журнал решений `15` §2 с датой. Отдельный `CHANGELOG.md` не заводим.
- **Вторая пара глаз:** правки канона и клиент-видимых текстов (лендинг, посты,
  оффер) проходят ревью (практика реализатор↔ревьюер).
- **Без новых файлов ради процесса** — cadence живёт в README и `15`, не в
  отдельном доке.
- **Реклама нутрициологии:** позиция «сопровождение, не медуслуга» — канон (`10`
  §3.2 / `COPY_VOICE`); статус законопроекта проверяется в еженедельном
  datestamp-чеке (п.4).

Путь чтения за ~30 мин:

   ведения статусов.

> основателя первым; ведение клиентов — позже). См. `25` § Bus-factor.
