# 32 · ПДн/RKN governance: релизный контур данных о здоровье

> **Статус:** source of truth / active runbook для ПДн/RKN gates. Задачи
> зеркалятся в `22`/`25`; детали доказательств и контур проверок — здесь.

Статус: **релизный блокер до активного сбора заявок и триалов**. Документ
раскладывает задачу `22` 1.9 в исполнимую систему: какие артефакты должны быть
созданы, что проверять перед релизом, что блокирует оплату и что блокирует
масштабирование.

> Не юридическое заключение. Это операционный контур, который снижает риск:
> вместо разовой подачи в РКН появляется доказуемая дисциплина обработки ПДн.
> Внешний юрист нужен перед публичным масштабированием и перед передачей данных
> новым обработчикам/сервисам.

---

## 0. Принцип

Канон по самому процессу ПДн —
`docs/legal/operator/pdn-compliance-operating-model.md`. Этот документ не
дублирует юридическую базу, а переводит её в release gates для маркетинга: что
блокирует заявки, триалы, оплаты, paid-scale и Ф1.

HEYS обрабатывает данные, связанные с состоянием человека: питание, вес, сон,
активность, цикл, самочувствие, стресс и другие health/recovery-сигналы. Поэтому
логика запуска такая:

1. **Сначала карта данных и правовые основания.**
2. **Потом РКН/согласия/защита.**
3. **Потом активный сбор заявок и триалы.**
4. **Платежи и платный трафик — только после повторной проверки получателей,
   целей и документов.**

Главный риск не в том, что HEYS честно регистрирует спецкатегорию. Главный риск
— чтобы продукт, лендинг, база, кураторы, согласия и запись РКН говорили разные
вещи.

### B. Data-change gate

Цель: новая фича не добавляет ПДн незаметно.

Перед релизом любой фичи задаются 5 вопросов:

1. Добавляется новая категория ПДн или новая health-сущность?
2. Меняется цель обработки?
3. Появляется новый получатель/обработчик: куратор, платежи, аналитика, бот, AI,
   рассылка?
4. Появляется трансграничная передача или зарубежный сервис?
5. Меняются сроки хранения, доступы, логи или способ удаления?

Если хотя бы один ответ «да» — выкладка блокируется до обновления:

- реестра данных;
- privacy/consent/user agreement, если затронуты пользовательские условия;
- РКН-уведомления/изменений, если меняются сведения реестра;
- технической схемы доступа/логирования;
- записи в `15` при изменении канона.

Acceptance:

| Проверка                                                                         | Статус | Что осталось                |
| -------------------------------------------------------------------------------- | ------ | --------------------------- |
| В pre-release чеклисте есть строка «ПДн data-change gate пройден»                | ✅     | применять при каждом релизе |
| Для каждого «да» есть ссылка на документ/задачу                                  | ✅     | поддерживать в review       |
| Спорное legal-изменение не закрывается самим исполнителем перед масштабированием | 🟡     | нужен внешний sign-off      |

### C. Согласия и доказательства

Цель: можно доказать, кто, когда и на что согласился.

Минимальный набор доказательств:

- тип согласия: `personal_data`, `health_data`, `marketing`, `payment_oferta`;
- версия документа;
- дата/время;
- `client_id`;
- способ подписи: checkbox после аутентификации / SMS позже;
- IP и user-agent;
- активность/отзыв;
- ссылка на архивную версию документа.

Acceptance:

| Проверка                                                                                   | Статус | Что осталось                                                                                 |
| ------------------------------------------------------------------------------------------ | ------ | -------------------------------------------------------------------------------------------- |
| Health consent отдельный от общего согласия                                                | ✅     | сверять версии документа                                                                     |
| Marketing consent необязательный и отделен от сервиса                                      | ✅     | не связывать с доступом к услуге                                                             |
| Отзыв согласия имеет понятный сценарий                                                     | ✅     | проверить в фактическом flow                                                                 |
| Новый пользователь без health consent не попадает в полноценный дневник/аналитику здоровья | ✅     | consent/proof smoke PASS: `missing` включает `health_data`, gate `valid=false` до подписания |

### D. Технический контур защиты

Цель: меры защиты совпадают с тем, что декларируется в `31` и privacy.

Минимум для R0:

- health-ключи шифруются или явно отмечены как gap;
- audit logs не хранят значения health-data в открытом виде;
- куратор видит только назначенных клиентов;
- Telegram-уведомления содержат только `lead_id`, без телефона/ФИО/health-data;
- GA4/Meta/зарубежные пиксели выключены;
- Yandex Metrica включается только через consent-gate;
- есть backup/restore и план удаления;
- есть список секретов и мест хранения: Lockbox/env, без секретов в репо.

Acceptance:

| Проверка                                                                                        | Статус | Что осталось                               |
| ----------------------------------------------------------------------------------------------- | ------ | ------------------------------------------ |
| Есть список фактически защищенных ключей и gap-list                                             | ✅     | обновлять при новых ключах                 |
| Есть access matrix: оператор / назначенный куратор / подменный куратор / техподдержка / система | ✅     | держать вместе с договорами кураторов      |

### F. Ежемесячная сверка

Цель: не допустить расхождения «в продукте одно, в документах другое».

Проверяются 6 слоев:

1. **Продукт:** какие поля реально собираются.
2. **БД/логи:** где эти поля реально лежат.
3. **Лендинг:** что обещаем и какие согласия берем.
4. **Документы:** privacy, health consent, user agreement, operator docs.
5. **РКН:** цели, категории субъектов/ПДн, получатели, трансгран.
6. **Кураторы/операции:** кто реально видит данные и куда их пересылает.

Acceptance:

| Проверка                                                                        | Статус | Что осталось                        |
| ------------------------------------------------------------------------------- | ------ | ----------------------------------- |
| Есть monthly audit note: дата, кто проверил, что изменилось, что открыть в `22` | ✅     | заполнить фактическую первую запись |
| Любой gap становится задачей в `22`, а решение — записью в `15`                 | ✅     | применять при сверке                |
| Перед масштабированием есть внешний legal sign-off по последней версии          | 🟡     | package готов; получить перед R2/R3 |

## 3. R1: до первых оплат

До включения реальных платежей нельзя просто «добавить ЮKassa». Нужно пройти
data-change gate:

- какие ПДн уходят в ЮKassa/ЮMoney;
- какие данные попадают в чеки и бухгалтерию;
- нужно ли обновить privacy, user agreement, payment consent;
- нужно ли обновить запись РКН;
- не попадают ли health-data в платежные events, metadata, webhook logs;
- где хранится связь платежа с клиентом и как она обезличивается при удалении.

Acceptance:

| Проверка                                                                      | Статус | Что осталось                                                                             |
| ----------------------------------------------------------------------------- | ------ | ---------------------------------------------------------------------------------------- |
| `heys-api-payments` готов и не пишет contact/health-data в платежные metadata | 🟡     | source payload/webhook tests PASS; подтвердить на live create/status/refund после deploy |
| Payment consent актуален                                                      | 🟡     | финально сверить перед включением оплат                                                  |
| Privacy/RKN проверены после фактической схемы платежей                        | 🟡     | пройти data-change gate после фактических YooKassa/receipt полей                         |
| Платежный webhook не раскрывает лишние ПДн в логах                            | 🟡     | локальный dedupe/metadata test PASS; нужен live webhook/log smoke                        |

## 4. R2/R3: до масштабирования

Перед платным трафиком, посевами и ростом выше ручного теста:

- внешний legal sign-off по
  [heys-r0-r2-signoff-package.md](../docs/legal/operator/heys-r0-r2-signoff-package.md):
  `31`, этот документ, privacy, health consent, user agreement, договор
  куратора, access matrix, incident playbook;
- ERID-процесс закрыт;
- кураторские договоры включают конфиденциальность, доступ только к назначенным

## 6. Что не решается этим документом

- Он не заменяет юриста.
- Он не определяет точный уровень защищённости ИСПДн без модели угроз.
- Он не разрешает медицинские claims в маркетинге.
- Он не разрешает зарубежные сервисы по умолчанию.

Документ отвечает только за одно: **перед релизом у HEYS есть понятная и
проверяемая система обработки ПДн, а не набор разрозненных обещаний.**
