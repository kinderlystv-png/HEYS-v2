# 38 · AI-ввод еды по фото / штрихкоду — R&D

> **Статус:** evidence/reference. Это закрытие R&D-решения; текущий AI-канон
> остаётся в `20`/`21`, живые задачи — в `22`.

Дата: 2026-06-14.

## Короткое решение

`4.3` закрывается как исследование: AI-ввод не выносить в обещание Pro-Ф0; для
будущего Self/Solo безопасный путь — сначала штрихкод и ручное подтверждение,
потом фото как черновик записи, который пользователь или куратор обязан
проверить.

Это не “AI считает еду”, а помощник для подготовки записи. Как автозаполнение в
форме: оно экономит время, но человек всё равно смотрит, правильно ли
подставлены поля.

## Что запускать первым

### 1. Barcode-first

Штрихкод даёт более надёжный старт, чем фото: он привязан к конкретному товару,
а не к догадке модели по картинке.

Рекомендуемый контур:

```text
PWA scan / manual barcode
→ backend lookup
→ Open Food Facts by barcode
→ normalize product_name + nutriments per 100g
→ user confirms grams / portion
→ save as draft or diary entry
```

Правила качества:

- не записывать найденный продукт автоматически;
- показывать источник и дату/качество данных, если поле доступно;
- если нет продукта или нутриентов — отправлять в ручной ввод;
- кэшировать ответ, но не считать Open Food Facts единственным источником
  истины;
- не отправлять в Open Food Facts ПДн пользователя.

Open Food Facts подходит как первый справочник: API поддерживает поиск товара по
barcode и выдаёт название/нутриенты. Важно: сама документация предупреждает, что
данные добровольные и без гарантий точности/полноты.

### 2. Photo as assisted draft

Фото использовать только как черновик:

```text
photo + optional text
→ vision model
→ candidate items + grams estimate + confidence + questions
→ user/curator confirms
→ entry saved
```

Минимальный JSON-контракт:

```json
{
  "items": [
    {
      "name": "rice, cooked",
      "grams": 180,
      "confidence": 0.68,
      "needs_clarification": false
    }
  ],
  "questions": ["Was there oil, sauce, or sugar?"],
  "warnings": ["Portion size is estimated from the image."]
}
```

Стоп-правила:

- не писать “точно считает по фото”;
- не делать автоматическую запись без подтверждения;
- не использовать фото-AI в Pro-Ф0 как замену вниманию куратора;
- не отправлять ФИО, телефон, email, диагнозы или комментарии куратора в
  vision-запрос;
- очищать EXIF до отправки/хранения.

## Провайдеры и данные

| Контур                      | Вывод                                                                                                                                                                                                                                   |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Open Food Facts             | Первый источник для barcode lookup. Подходит для MVP/R&D, но требует ручной проверки и fallback, потому что база открытая и добровольная.                                                                                               |
| GS1 Digital Link / GS1 APIs | Полезно как стандарт и enterprise-направление, но это не бесплатная полная база нутриентов для старта. GS1 US API — подписочный/enterprise-контур.                                                                                      |
| OpenAI vision               | Технически умеет принимать изображения, но для HEYS это не основной production-путь из-за ПДн/трансграничного контура. Можно использовать только для внутреннего non-prod исследования на обезличенных тестах, если юр-контур разрешит. |
| RF-native vision            | Предпочтительный production-кандидат для фото-R&D: Yandex/GigaChat или другой российский провайдер после отдельного privacy review.                                                                                                     |

## Комплаенс

Фото еды в связке с дневником, весом, целью и аккаунтом считаем персональными
данными или минимум health-adjacent данными.

Перед публичным AI нужны:

- обновление согласия и политики: фото, питание, AI-обработка, провайдеры, сроки
  хранения, удаление;
- отдельная запись в реестре данных;
- DPA/договор с провайдером;
- срок хранения исходного фото после распознавания;
- логирование факта AI-обработки без содержимого фото;
- пользовательский текст “результат требует проверки”.

## Go / no-go критерии

Запускать barcode beta, если:

- lookup возвращает продукт и нутриенты на тестовой выборке популярных товаров;
- есть ручной fallback за один экран;
- нет записи без подтверждения пользователя;
- источники и сомнения видны в UI.

Запускать photo beta, если:

- на архиве реальных кейсов модель даёт полезный черновик, а не иллюзию
  точности;
- у результата есть confidence, вопросы и ручное подтверждение;
- legal/privacy review закрыт;
- стоимость одного запроса укладывается в экономику Self/Solo;
- UX не переносит ответственность с пользователя/куратора на модель.

Не запускать, если:

- маркетинг просит обещание “сфотографировал — всё точно посчиталось”;
- нет российского или юридически разрешённого контура обработки фото;
- модель часто ошибается в масле, соусах, смешанных блюдах и граммах;
- функция ухудшает чистоту данных Pro.

## Product decision

Для текущего запуска:

- Self: ручной дневник и личная база продуктов; AI не обещать.
- Pro/Pro+: ручной ввод куратора остаётся частью ценности.
- После Ф0: barcode beta как первый безопасный Self/Solo R&D.
- Фото-AI: только после barcode и только как проверяемый черновик.

Формулировка для будущего backlog:

```text
Barcode lookup and photo draft help prepare a food entry faster, but HEYS saves
nutrition data only after user or curator confirmation.
```
