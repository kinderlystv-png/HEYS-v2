# HEYS — спецификация измеримости (без трансграничной передачи)

> Прикладной слой. Дата: 2026-06-09. Аналитика намеренно урезана под 152-ФЗ
> (GA4/Meta Pixel = `null`, см. бриф/`10`). Чтобы не оптимизировать трафик
> вслепую — нужна **собственная измеримость**: метки + серверная Yandex
> Metrica + простая CRM. Всё в РФ-контуре, без ПДн на зарубежные сервисы.

---

## 1. Принцип

Каждый лид помечен источником на входе; считаем **сквозную воронку по
источнику** до оплаты и продления. Метки (UTM/промокоды) — не персональные
данные, передавать их безопасно. ПДн (имя/телефон) живут в РФ-CRM, на сторонние
сервисы не уходят.

---

## 2. UTM-схема (для всех ссылок)

Единые соглашения:

| Параметр       | Значения                                              | Пример           |
| -------------- | ----------------------------------------------------- | ---------------- |
| `utm_source`   | `yandex`, `telegram`, `vk`, `seed_<канал>`, `organic` | `seed_zojmama`   |
| `utm_medium`   | `cpc`, `post`, `tg_ads`, `seeding`, `bio`             | `seeding`        |
| `utm_campaign` | кампания/оффер                                        | `first15_launch` |
| `utm_content`  | креатив/объявление                                    | `hook_evening`   |

Telegram-бот: источник передаём через **deep link** `t.me/<bot>?start=src_<код>`
(параметр `start` логируется при первом входе). Один источник = один
`start`-код.

---

## 3. Промокоды / кодовые слова

Где UTM теряется (устные посевы, офлайн, репосты) — **уникальный промокод** на
канал/интеграцию. Таблица соответствия канал → код ведётся в CRM. Промокод
называет лид при записи → источник восстановлен.

---

## 4. Серверная Yandex Metrica (события-цели)

Серверная отправка (Measurement Protocol) из бэкенда, **без трансграничной
передачи**. Цели/события воронки:

| Событие         | Когда                                | Параметры                   |
| --------------- | ------------------------------------ | --------------------------- |
| `lead`          | старт бота / заявка                  | source, campaign            |
| `quiz_complete` | завершил квиз                        | source, segment (тип срыва) |
| `week_request`  | запросил неделю                      | source, готовность          |
| `trial_active`  | активный триал (≥4 дня, ≥12 приёмов) | source, segment             |
| `payment`       | первая оплата                        | source, tariff              |
| `renewal`       | продление                            | tariff                      |

---

## 5. CRM (минимальная, на старте — таблица)

Поля: `lead_id` · дата · **источник** (utm/промокод) · тип срыва/сегмент ·
статус (`лид → триал → активный → оплатил → продлил → отвалился`) · тариф ·
куратор · комментарий. На Фазе 0 хватит простой таблицы; данные — в РФ.

---

## 6. Отчёт воронки (еженедельно → KPI-трекер дашборда)

По каждому источнику: лиды · CPL · `quiz→неделя` · `неделя→оплата` · CAC ·
LTV:CAC. Сводно — в лист «KPI-трекер» `00_Сводная_панель.xlsx`. Решение о
масштабировании канала — только по этим цифрам, не «на глаз».

---

## 7. «Откуда узнали» — резерв

На шаге записи — необязательный вопрос «как узнали о нас?» (варианты + свой
текст). Страховка на случай потери меток (репосты, сарафан).

---

## 8. Чего НЕ делать

- Не ставить GA4 / Meta Pixel (трансграничная передача → 152-ФЗ).
- Не слать события с ПДн (имя/телефон) на зарубежные сервисы.
- UTM/промокоды/обезличенные события — можно: это не ПДн.
